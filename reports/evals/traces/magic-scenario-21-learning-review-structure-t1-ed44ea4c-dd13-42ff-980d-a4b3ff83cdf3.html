# Learning Review: Seasonal Pricing Cache Staleness Incident (2026-03-12) ## Situation Snapshot On 2026-03-12, a cache invalidation refactor (pricing-service v2.4.1, PR #1847) was deployed to production by Shaq and reviewed by Kobe. The change altered how TTL values are computed for category-specific cache entries, moving from config-driven TTL lookup to call-site TTL passing. A missing wiring for the seasonal category's 15-minute TTL override caused Redis to retain stale promotional prices. 847 customer transactions were affected over a 20-minute window (14:22–14:42 UTC) before Pippen's price divergence alert triggered and the deployment was rolled back. ## Timeline | Time (UTC) | Event | What was known | Decision | |---|---|---|---| | Pre-deploy | PR #1847 opened as "performance optimization" for cache invalidation | Ticket scoped as perf work; Bird not consulted | Shaq implemented, Kobe reviewed — reasonable given the ticket's stated scope | | Pre-deploy | Kobe reviews PR | Unit tests pass; standard product path looks correct | Approved. Seasonal TTL override in CategoryConfig was not checked — the PR description framed this as a generic TTL plumbing change, not a category-behavior change | | Pre-deploy | Shaq tests in staging | Staging exercises "standard" category successfully | Deployed to production. Seasonal products were absent from the staging fixture set, so there was no signal that anything was wrong | | 14:22 | Deployment to production | All health checks green | Normal deploy process followed | | 14:22–14:38 | Customers purchasing seasonal items are charged previous day's promotional prices | No signal yet — the system is serving cached values that look structurally valid | No action possible; nobody knows there is a problem | | 14:38 | Pippen's price divergence alert fires (5% threshold breached) | Alert identifies divergence between pricing-service responses and source-of-truth feed | On-call begins investigation | | 14:42 | Rollback completed | Root deployment identified as v2.4.1 | Rollback executed; prices return to correct values | ## Contributing Factors 1. **Ticket scoping excluded domain expertise.** The cache invalidation work was categorized as a "performance optimization," which meant Bird — the domain authority on pricing rules — was never consulted during specification. The seasonal category's 15-minute TTL constraint is a domain rule with financial consequences, but it was invisible to the people working the ticket. 2. **The seasonal TTL rule lived only in CategoryConfig, not in acceptance criteria.** The rule that seasonal TTL must not exceed 15 minutes during promotional windows is encoded in a config enum but was not surfaced as a testable constraint on the ticket. There was no mechanism to ensure that a refactor touching TTL computation would be checked against category-specific domain rules. 3. **Staging fixture set did not cover seasonal products.** Shaq tested the change against the "standard" category in staging, which exercised the happy path. Seasonal products were absent from the staging data, so the misconfigured TTL path was never executed before production. This made sense at the time — the ticket appeared to be a generic plumbing change, and testing the standard path seemed sufficient. 4. **Code review focused on the changed path, not on what was no longer wired.** Kobe verified that the new call-site TTL pattern worked correctly for the standard product path and that unit tests passed. The seasonal category's TTL override in CategoryConfig was not checked because the PR did not visibly touch seasonal logic — the bug was in what the new code *failed to read*, not in what it explicitly changed. Reviewing for omissions is structurally harder than reviewing for errors. 5. **No automated validation of category TTLs against domain rules.** There was no integration test or contract test that asserted "for each category with a TTL override in CategoryConfig, the cache layer must respect that override." This meant the safety net for this class of bug was entirely manual review. 6. **Alert detection latency of 16 minutes.** The price divergence alert threshold (5% divergence) took 16 minutes to breach. During that window, 847 transactions completed at incorrect prices. The alert worked as designed — this is not a failure — but the detection window is worth examining given the financial sensitivity of pricing errors. ## Learnings - **We now know that cache TTL is a domain-rule-carrying parameter, not just a performance knob**, which means any change to TTL computation must be treated as a potential pricing rule change and reviewed with domain context, regardless of how the ticket is categorized. - **We now know that our staging fixture set has categorical blind spots**, which means staging validation only proves the paths that staging data covers. For changes that affect category-specific behavior, we need either broader fixture coverage or an explicit checklist of which categories were tested. - **We now know that "performance optimization" scoping can silently exclude domain review**, which means ticket categorization has downstream consequences for who reviews the work. A ticket that touches domain-rule-adjacent logic needs domain input even if its *intent* is purely technical. - **We now know that code review is structurally weak at catching omissions** (things the code should read but doesn't), which means we need automated guards for invariants like "every CategoryConfig TTL override must be consumed by the cache layer." - **We now know that 16 minutes of undetected pricing divergence can affect ~850 transactions**, which means our current alert threshold and polling interval establish the blast radius floor for pricing bugs. Whether that floor is acceptable is a business decision, not a technical one. ## Action Items 1. `[PREVENT]` **Add a contract test that validates category-specific cache TTLs against CategoryConfig domain rules.** For every category with a TTL override in CategoryConfig, assert that the cache layer produces a TTL matching the config value. — **Owner: Shaq** — **Due: 2026-03-28** 2. `[PREVENT]` **Expand staging fixture set to include at least one product per pricing category** (standard, seasonal, clearance, etc.). — **Owner: Pippen** — **Due: 2026-04-04** 3. `[PREVENT]` **Add a domain-review gate for tickets touching cache TTL, pricing logic, or CategoryConfig.** Any PR modifying these paths requires Bird's sign-off, regardless of ticket category. Implement as a CODEOWNERS rule or equivalent. — **Owner: Coach K** — **Due: 2026-03-31** 4. `[DETECT]` **Evaluate tightening the price divergence alert threshold or polling interval.** Current: 5% divergence. Determine whether a lower threshold or faster poll (e.g., 2-minute sampling) would meaningfully reduce the transaction exposure window without generating false positives. — **Owner: Pippen** — **Due: 2026-04-04** 5. `[PROCESS]` **Add a "Domain Rules Affected" section to the PR template for pricing-service.** Require authors to explicitly list which domain rules (if any) are touched or could be affected by the change. An empty list is fine — the point is to force the question. — **Owner: Kobe** — **Due: 2026-03-31** 6. `[MITIGATE]` **Implement a canary deployment strategy for pricing-service** that compares cache-served prices against the source-of-truth feed for a configurable window before full rollout. — **Owner: Pippen + Shaq** — **Due: 2026-04-11** ## Preserving the Learning - **ADR: "Cache TTL as Domain Rule Carrier"** — Document the decision that TTL values in the pricing service are domain-constrained, not arbitrary performance parameters. Reference the seasonal 15-minute rule as the canonical example. This ADR should live in `docs/adrs/` and be linked from the pricing-service README. — **Owner: Magic (me)** - **Domain Rule Registry entry** — Bird to add the seasonal TTL constraint to a machine-readable domain rule registry (or create one if it doesn't exist) so that future contract tests can be generated from it rather than hand-written. — **Owner: Bird** - **Code review checklist addition** — Kobe to add "For changes that alter how configuration values are consumed: verify all configuration sources are still wired" to the pricing-service review checklist. — **Owner: Kobe** - **Staging fixture coverage policy** — Pippen to document the requirement that staging must cover all pricing categories, with a CI check that validates fixture category coverage against the CategoryConfig enum. — **Owner: Pippen**
