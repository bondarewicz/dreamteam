# Learning Review: Seasonal Pricing Cache Staleness Incident (2026-03-12) ## Situation Snapshot On 2026-03-12, a cache invalidation refactor (pricing-service v2.4.1, PR #1847) was deployed by Shaq and reviewed by Kobe. The change altered how TTL values are computed for category-specific Redis cache entries, moving from config-driven TTL lookup to call-site TTL passing. A misconfigured TTL override caused the "seasonal" product category to retain stale promotional prices, resulting in 847 customer transactions charged at incorrect (previous-day) prices over a 20-minute window (14:22–14:42 UTC). Pippen's price divergence alert detected the issue at 14:38 UTC and rollback was completed four minutes later. ## Timeline | Time (UTC) | Event | What was known | Decision / Action | |---|---|---|---| | Pre-deploy | PR #1847 created, scoped as "performance optimization" for cache invalidation | Ticket framed as perf work, not pricing-rule work. Bird not consulted. | Shaq implemented; Kobe reviewed and approved. | | ~Pre-deploy | Staging validation performed | Shaq tested against "standard" category fixtures. Seasonal products absent from staging data. | Passed staging — no signal that seasonal path was broken. | | 14:22 | Deployment to production | Green CI, approved review, staging pass — all signals positive. | Deploy proceeded normally. | | 14:22–14:38 | Stale seasonal prices served from Redis | No alert yet — divergence had not crossed the 5% threshold long enough to trigger. | No action; no one aware of the issue. | | 14:38 | Pippen's price divergence alert fires | Alert indicated pricing-service responses diverging from source-of-truth feed. | Incident response initiated; investigation pointed to the new deployment. | | 14:42 | Rollback completed | Root deployment identified as v2.4.1. | Rollback executed; stale cache entries expired or were flushed. | ## Contributing Factors No single cause produced this incident. The following independent factors each contributed, and removing any one of them would likely have prevented or significantly reduced the impact: 1. **Ticket scoping excluded domain review.** The cache invalidation ticket was categorized as a "performance optimization." This framing made it reasonable for Bird — the domain authority on pricing rules — not to be consulted. The seasonal TTL constraint (max 15 minutes during promotional windows) was therefore invisible to both the implementer and the reviewer. This was a sensible scoping decision at the time: most cache TTL changes genuinely are pure performance work. 2. **The seasonal TTL domain rule was encoded only in CategoryConfig, not in acceptance criteria or tests.** The rule that seasonal TTL must not exceed 15 minutes during promotions existed as configuration, but had no corresponding automated assertion. There was no test that validated "for each category, the effective cache TTL must not exceed the CategoryConfig-defined maximum." The rule was correct in code but not defended by any verification layer. 3. **Staging fixture set did not include seasonal products.** Shaq tested against the "standard" category in staging, which exercised the new call-site TTL pattern correctly. Seasonal products were absent from fixtures. This made staging validation pass cleanly while leaving the broken path completely unexercised. The fixture gap was not previously identified as a risk because prior changes had not altered TTL computation paths. 4. **Code review focused on the standard path.** Kobe verified that the new TTL logic was correct for standard products and that unit tests passed. The CategoryConfig override for seasonal products was not checked against the new call-site pattern. This is understandable: the PR description and ticket framing both pointed toward a generic performance change, not category-specific behavior. The review followed the framing it was given. 5. **Price divergence alert had a detection lag.** Pippen's alert threshold (5% divergence) took 16 minutes to fire after deployment. This is a reasonable threshold for avoiding false positives, but it meant the incident window extended well beyond what a tighter or category-aware alert might have allowed. There was no category-specific pricing integrity check that could have caught the issue faster. ## Learnings - **We now know that cache TTL changes are implicitly pricing-rule changes for any category with a domain-mandated TTL ceiling.** This means any ticket that touches TTL computation — regardless of how it is scoped — must be flagged for domain review when it could affect categories with TTL constraints. - **We now know that the seasonal category's TTL invariant (<=15 min during promos) was a load-bearing domain rule with no automated defender.** This means domain rules that live only in config enums are invisible to CI and code review unless they are also expressed as executable tests. - **We now know that staging fixture coverage has a direct security-of-pricing implication.** The absence of seasonal products from staging fixtures was not a known risk until this incident. This means fixture coverage should be treated as a form of test coverage — gaps should be visible and tracked. - **We now know that ticket scoping ("performance optimization" vs. "pricing rule change") directly controls which agents review the work.** A mis-scoped ticket is not negligence; it is an information routing failure. This means we need a lightweight mechanism to detect when a "perf" change touches domain-sensitive paths, independent of how the ticket is labeled. - **We now know that 16 minutes to detection is the current floor for price divergence incidents.** For time-sensitive promotional pricing, this detection window may be too wide. This means we should evaluate whether category-aware or TTL-specific alerts can reduce time-to-detection for pricing integrity issues. ## Action Items 1. `[PREVENT]` **Add a domain-rule integration test**: Create an automated test that, for every product category in CategoryConfig, asserts that the effective cache TTL does not exceed the category's configured maximum. This test must run in CI on every PR that touches cache or TTL code. **Owner: Shaq. Due: 2026-03-28.** 2. `[PREVENT]` **Expand staging fixtures to include all pricing categories**: Add seasonal (and any other category with domain-specific TTL rules) to the staging fixture set. Validate fixture coverage against the CategoryConfig enum so new categories cannot be added without corresponding fixtures. **Owner: Shaq. Due: 2026-03-31.** 3. `[DETECT]` **Add a category-aware pricing integrity alert**: Supplement the existing 5% divergence alert with a faster, narrower check: for categories with active promotional windows, alert on any TTL-exceeding-config condition within 2 minutes of deployment. **Owner: Pippen. Due: 2026-04-04.** 4. `[PROCESS]` **Introduce a "domain-sensitive paths" annotation for code review**: Maintain a lightweight list of code paths (e.g., TTL computation, price calculation, discount application) that require domain review regardless of ticket scoping. When a PR touches an annotated path, the review checklist should prompt the reviewer to verify domain constraints or request Bird's input. **Owner: Kobe, with Bird providing the initial path list. Due: 2026-04-07.** 5. `[PROCESS]` **Require domain review for any ticket touching CategoryConfig-governed behavior**: Update the ticket workflow so that changes affecting category-specific configuration — even when scoped as performance work — are automatically flagged for Bird's review. **Owner: Coach K (process). Due: 2026-04-07.** 6. `[MITIGATE]` **Evaluate canary deployment for pricing-service**: Investigate whether a canary deployment strategy (serving a small percentage of traffic before full rollout) would have limited the blast radius from 847 transactions to a smaller number. **Owner: Pippen. Due: 2026-04-14.** ## Preserving the Learning - **ADR: "Cache TTL Changes Require Domain Review When Category Constraints Exist"** — Document the decision that TTL computation changes are domain-sensitive, not purely operational. Include the rationale from this incident. **Owner: Magic. To be committed by 2026-03-28.** - **Domain Rule Catalog Entry**: Add the seasonal TTL invariant (`<= 15 min during active promo windows`) to a formalized domain rule catalog (or CLAUDE.md if no catalog exists), with a pointer to the defending test. This ensures the rule is discoverable outside of CategoryConfig source code. **Owner: Bird. Due: 2026-03-31.** - **Review Checklist Update**: Add "Does this PR touch a domain-sensitive path?" to the standard code review checklist, with a link to the annotated path list. **Owner: Kobe. Due: 2026-04-07.** - **Fixture Coverage Report**: Add a CI step that reports which product categories are present in staging fixtures vs. CategoryConfig, failing if coverage drops below 100%. This makes the fixture gap that enabled this incident structurally impossible to recur. **Owner: Shaq. Due: 2026-03-31.**
