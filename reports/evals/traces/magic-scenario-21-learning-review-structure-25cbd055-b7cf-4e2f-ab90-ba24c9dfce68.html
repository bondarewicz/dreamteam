# Learning Review: Seasonal Pricing Cache Staleness Incident (2026-03-12) ## Situation Snapshot On 2026-03-12, a cache invalidation refactor (pricing-service v2.4.1, PR #1847) was deployed to the product pricing service. The change altered how TTL values are computed for category-specific cache entries, moving from config-driven TTL lookup to call-site-passed TTL. A missed wiring for the "seasonal" category's 15-minute TTL override caused Redis to retain stale promotional prices. 847 customer transactions over 20 minutes were affected — customers purchasing seasonal items were charged previous-day promotional prices. The deployment was rolled back at 14:42 UTC after Pippen's price divergence alert fired at 14:38 UTC. ## Timeline - **Pre-deploy (days prior)**: Ticket scoped as "performance optimization" for cache invalidation. Because it was categorized as a performance change, Bird (domain authority) was not consulted during spec or review. Shaq implemented the change; Kobe reviewed and approved. - **14:22 UTC**: Shaq deploys pricing-service v2.4.1 to production. At this moment, the team believes the change is a safe refactor — staging tests passed against the "standard" category, which is the most common path. - **14:22–14:38 UTC (16 minutes)**: Seasonal category cache entries are being served with an incorrect (longer) TTL. Customers hitting seasonal products see stale prices from the previous day's promotional batch. No one is aware yet — the system looks healthy from standard metrics. - **14:38 UTC**: Pippen's price divergence alert fires. The alert detects >5% divergence between the pricing-service response and the source-of-truth price feed. This is the first signal that something is wrong. - **14:38–14:42 UTC (4 minutes)**: Team investigates, identifies the cache TTL as the culprit, and rolls back to v2.4.0. Rollback completes at 14:42 UTC. - **Post-incident**: Bird surfaces the domain rule that seasonal TTL must not exceed 15 minutes during active promotional windows — a rule that was not encoded in the ticket's acceptance criteria. ## Contributing Factors 1. **Ticket scoping excluded domain review.** The cache invalidation change was categorized as a "performance optimization," which meant Bird was not consulted during spec. The change actually touched pricing-rule-adjacent behavior (category-specific TTL), but this was not recognized at scoping time. This made sense given the framing — performance work typically does not alter business rules. 2. **Staging fixture set did not include seasonal products.** Shaq tested against the "standard" category in staging. Seasonal products were absent from the staging fixture data. The test passed, reinforcing confidence that the change was safe. This is a reasonable testing choice if you believe the code path is category-agnostic — which the refactor appeared to be. 3. **The TTL domain rule was implicit in code, not in acceptance criteria.** The seasonal category's 15-minute TTL ceiling exists as a value in the `CategoryConfig` enum, but it was never expressed as a testable acceptance criterion on the ticket. The rule's business rationale (promotional prices are time-sensitive) was known to Bird but not surfaced to the implementer or reviewer. 4. **Code review focused on the changed path, not the unchanged paths it displaced.** Kobe verified that the new call-site TTL pattern worked correctly for the standard product path and that unit tests passed. He did not check whether the new pattern preserved the seasonal category's TTL override. This is a natural review heuristic — verify what changed — but the risk here was in what the change *stopped doing* (reading from CategoryConfig). 5. **No automated verification of category-specific TTL compliance.** There was no integration or contract test asserting that each category's effective cache TTL matches its `CategoryConfig` domain rule. The domain rule existed only as a config value, not as a tested invariant. 6. **Detection latency of 16 minutes.** The price divergence alert's 5% threshold and evaluation window meant 16 minutes elapsed before the anomaly was detected. This is not unreasonable for a statistical anomaly detector — it needs enough data points to distinguish signal from noise — but it meant the blast radius grew for the full window. ## Learnings - **We now know that "performance optimization" tickets can silently cross domain boundaries**, which means any change to cache TTL, invalidation, or data freshness logic should be flagged for domain review regardless of how the ticket is categorized. The category of the ticket is not a reliable proxy for the category of risk. - **We now know that the seasonal TTL rule is a critical domain invariant that exists only in config, not in tests**, which means it can be silently violated by any refactor that changes how config values are consumed. Domain rules that are only encoded as config values are indistinguishable from tuning parameters to someone unfamiliar with the domain. - **We now know that staging fixture coverage has a blind spot around seasonal products**, which means any change touching category-specific behavior cannot be validated in staging without fixture updates. A test that passes against an incomplete fixture set provides false confidence. - **We now know that Pippen's price divergence alert works and catches this class of issue**, which means the observability layer is functioning as designed. The 16-minute detection window is the current cost of statistical confidence — faster detection would require either tighter thresholds (risking false positives) or a different detection approach (e.g., deterministic TTL assertion). - **We now know that code review heuristics that focus on "what changed" can miss risks in "what the change displaced"**, which means refactors that move responsibility (e.g., from config lookup to call-site passing) need explicit reviewer attention on whether all prior consumers of the old path are preserved. ## Action Items - `[PREVENT]` **Add domain review trigger for cache/freshness changes.** Any ticket that modifies cache TTL, invalidation logic, or data freshness behavior must be tagged for Bird's review during spec, regardless of whether the ticket is labeled "performance" or "feature." *Owner: Coach K. These tests should fail if a new code path bypasses the config lookup. *Owner: Shaq. By: 2026-04-07.* - `[DETECT]` **Add seasonal products to the staging fixture set.** Ensure staging data includes representative products for every pricing category, especially seasonal. Validate fixture completeness as part of the staging environment's health check. *Owner: Pippen. By: 2026-04-07.* - `[MITIGATE]` **Evaluate tighter price divergence alert for known promotional windows.** During active promotional windows, consider a lower divergence threshold or shorter evaluation window to reduce detection latency. Trade-off: may increase false positive rate outside promotional windows. *Owner: Pippen. By: 2026-04-14.* - `[PROCESS]` **Add "displaced behavior" checklist item to code review.** When a PR moves responsibility from one location to another (e.g., config lookup to call-site), the reviewer should explicitly verify that all prior consumers and overrides of the old location are preserved. *Owner: Kobe.
