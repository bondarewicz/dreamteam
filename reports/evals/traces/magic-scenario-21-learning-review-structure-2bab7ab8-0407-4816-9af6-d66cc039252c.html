# Learning Review: Stale Seasonal Pricing Incident (2026-03-12) ## Situation Snapshot On 2026-03-12, a cache invalidation change to the product pricing service (v2.4.1) introduced a misconfigured TTL for the "seasonal" product category, causing Redis to retain stale promotional prices. 847 customer transactions over a 20-minute window (14:22–14:42 UTC) were charged previous-day promotional prices. The issue was detected by an automated price divergence alert at 14:38 UTC and resolved via rollback at 14:42 UTC. Shaq deployed the change; Kobe reviewed and approved it; Bird was not consulted; Pippen's alerting detected the anomaly. ## Timeline | Time (UTC) | Event | What was known | Decision made | |---|---|---|---| | Pre-deploy | PR #1847 scoped as "performance optimization" — refactors TTL computation to pass TTL from call site instead of reading from CategoryConfig | Ticket framed as perf work; no pricing-rule implications surfaced | Bird not consulted; no domain review requested — reasonable given the ticket framing | | Pre-deploy | Kobe reviews PR #1847 | Unit tests pass; standard product path looks correct; seasonal category's TTL override exists in CategoryConfig but its relevance to this change is not obvious from the diff | Approved — the change appeared mechanically correct for the paths visible in the tests | | Pre-deploy | Shaq tests in staging with "standard" category | Staging fixture set does not include seasonal products | Shaq verifies the happy path; absence of seasonal fixtures is not flagged because the ticket is scoped to general cache behavior, not category-specific rules | | 14:22 | Deployment to production | All gates green | Deploy proceeds | | 14:22–14:38 | Seasonal products served with stale (previous-day) promotional prices | No one is aware; no immediate signal | Customers transact at incorrect prices | | 14:38 | Pippen's price divergence alert fires (5% threshold breached) | Alert surfaces that pricing-service responses diverge from source-of-truth feed | Incident response begins | | 14:42 | Rollback completed | Stale cache entries begin expiring; correct prices resume | 20-minute exposure window closed | ## Contributing Factors 1. **Ticket framing obscured domain implications.** The cache invalidation change was scoped as a "performance optimization." This framing made it invisible to the domain review process. The refactor actually changed *which code path determines TTL* — a behavioral change with pricing-rule consequences — but the ticket's label did not reflect this. No one was negligent here; the framing was locally reasonable because the primary *intent* was performance. 2. **Seasonal category TTL rule was not encoded as an acceptance criterion.** The domain rule ("seasonal TTL must not exceed 15 minutes during active promotional windows") lived only in the CategoryConfig enum and in Bird's domain knowledge. It was not expressed as a testable acceptance criterion on the ticket. Without that criterion, neither the implementer nor the reviewer had a specification to check against. 3. **Staging fixture set lacked seasonal product coverage.** Shaq tested against staging, which contained only "standard" category products. The absence of seasonal fixtures was not unusual — staging environments are commonly a subset of production — but it meant the one category with a strict TTL constraint was the one category not exercised. 4. **Code review focused on mechanical correctness, not domain invariant preservation.** Kobe verified that the new call-site TTL pattern was correctly implemented for the standard path and that unit tests passed. The review did not include a check of whether all category-specific TTL overrides were still honored. This is understandable: the diff was readable and the unit tests green. The CategoryConfig override is a domain concern that does not announce itself in a performance-focused diff. 5. **No automated test validates category TTLs against domain rules.** There was no integration or contract test asserting that cache TTLs for each category match the values specified in CategoryConfig. This meant the domain rule could be silently violated with no test failure. 6. **Detection latency: 16 minutes from deploy to alert.** Pippen's price divergence alert uses a 5% threshold, which is appropriate for catching meaningful pricing errors without false positives. However, the 16-minute detection window allowed a significant number of transactions (847) to proceed at incorrect prices before the anomaly became visible. ## Learnings - **We now know that TTL changes are pricing-rule changes, not just performance changes**, which means any ticket that modifies cache TTL computation for category-specific entries must be flagged for domain review regardless of how the ticket is scoped. - **We now know that the seasonal category has a hard domain constraint (15-minute max TTL during promos) that is invisible in code review unless the reviewer already knows to look for it**, which means domain invariants that live only in config enums need to be surfaced as explicit, testable assertions — not just inline constants. - **We now know that our staging fixture set is a silent risk vector**, which means the absence of a product category from staging is not a neutral fact; it is an untested code path. Fixture coverage gaps should be treated as test coverage gaps. - **We now know that code review alone cannot catch domain-rule violations when the reviewer is not primed to look for them**, which means our process needs a mechanism to route changes to the right domain expert based on what the code *touches*, not just how the ticket is *labeled*. - **We now know that 16 minutes of undetected price divergence at current transaction volume translates to ~847 affected transactions**, which means our detection-to-rollback window has a concrete, measurable cost that we can use to justify investment in faster detection. ## Action Items 1. `[PREVENT]` **Add a contract test: category TTLs vs. CategoryConfig domain rules.** For every category defined in CategoryConfig, assert that the TTL applied by the cache invalidation logic matches the configured value. Owner: **Shaq**. Due: 2026-03-28. 2. `[PREVENT]` **Add seasonal products to the staging fixture set.** Ensure every product category with domain-specific pricing rules has at least one representative product in staging. Owner: **Shaq**. Due: 2026-03-28. 3. `[PROCESS]` **Introduce a "domain-touch" label for PRs that modify cache TTL, pricing logic, or category config.** Any PR with this label requires Bird's sign-off before merge, regardless of ticket scope. Owner: **Coach K** (process), **Kobe** (enforcement in review checklist). Due: 2026-04-04. 4. `[DETECT]` **Reduce price divergence alert detection latency.** Investigate lowering the alert evaluation interval or adding a deployment-correlated canary check that compares pricing-service output against the source-of-truth feed immediately after deploy. Target: detection within 5 minutes. Owner: **Pippen**. Due: 2026-04-11. 5. `[MITIGATE]` **Add a staged rollout for pricing-service deployments.** Route a small percentage of traffic to the new version first (canary deployment) so that pricing divergence affects fewer transactions before detection. Owner: **Pippen**. Due: 2026-04-18. 6. `[PROCESS]` **Audit existing CategoryConfig domain rules and express each as a testable acceptance criterion in the backlog.** This is a one-time sweep to close the gap between implicit domain knowledge and explicit test coverage. Owner: **Bird** (rules), **Magic** (documentation).
