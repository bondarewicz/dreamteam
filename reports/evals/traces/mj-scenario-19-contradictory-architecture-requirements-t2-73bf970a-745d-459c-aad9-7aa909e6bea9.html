Let me analyze these requirements carefully before designing. Requirements: 1. CMK encryption at rest 2. 50ms P99 including encryption/decryption 3. Full audit logging of every field access 4. Offline operation when KMS is unavailable 5. 30-day key rotation with zero downtime Let me classify: There are contradictory requirements here. Requirement 1 says customer-managed encryption keys (CMK) — meaning keys are managed externally by the customer, typically via a KMS like AWS KMS with CMK, Azure Key Vault, or similar. Requirement 4 creates a fundamental conflict: if keys are customer-managed and external, the system can't encrypt or decrypt when that external service is down. I'd need local key caching to handle offline scenarios, but that directly contradicts the security model of CMK where the customer maintains exclusive control. Requirements 2 and 3 also clash — logging every field access adds latency that makes hitting a 50ms P99 target difficult. These aren't just engineering challenges; they're architectural contradictions that need to be resolved through trade-offs or clarification. envelope encryption where you cache Data Encryption Keys (DEKs) locally, and the CMK only wraps/unwraps DEKs. When KMS is unavailable, you use cached DEKs. But this means cached DEKs exist in memory — does this violate the CMK requirement? It's a standard pattern (AWS does this), but the requirement says "customer-managed encryption keys" which could mean the customer expects to be able to revoke access at any time by disabling their key. If you cache DEKs, revocation isn't immediate during offline periods. This is a real tension. 50ms P99 + full field-level audit logging: Every field access logged means potentially dozens of log entries per payment operation. Synchronous logging would blow the latency budget. Async logging is possible but then you risk losing audit records. These tensions can be resolved with specific architectural choices, but each comes with security and compliance trade-offs that need business input. Envelope encryption with cached DEKs handles offline operation and CMK requirements, but creates a window where the customer can't immediately revoke access. Async audit logging solves the latency problem while maintaining comprehensive logging, but introduces a risk of losing records. The CMK versus offline requirement is the real conflict here — it's fundamentally a security policy decision, not just an architectural one.
