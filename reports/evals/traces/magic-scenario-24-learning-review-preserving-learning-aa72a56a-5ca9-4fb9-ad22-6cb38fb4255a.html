# Learning Review: FCA Audit Trail Non-Compliance (2026-01-19) ## 1. Situation Snapshot On 2026-01-19, an FCA audit flagged that six weeks of order state transition records for regulated instruments were non-compliant with COBS 11.2A audit trail requirements. The root issue was a semantic gap between MJ's event-sourced architecture (asynchronous projection to read models) and Bird's domain rule requiring synchronous, immediately-durable state transitions committed to the compliance record store before client confirmation. Four specific audit queries returned stale state due to projection lag. The compliance gap affected all regulated instrument order transitions processed since the architecture went live, and the team only learned of it through the regulator — not through internal detection. ## 2. Timeline **~2025-09-19 (approximately 4 months before incident):** MJ designs event-sourced order state transition architecture. ADR-014 is written and submitted for review. MJ interprets "durable" in Bird's domain rule in the infrastructure sense (fsync to disk), not the regulatory sense (synchronous commitment to a compliance-grade store before client confirmation). This interpretation was reasonable from a pure engineering perspective — the event log *is* durable in the database sense. **~2025-09-19 to ~2025-09-26 (ADR review period):** Kobe reviews ADR-014 against his standard checklist: scalability, failure modes, operational complexity, security. The review passes. Bird is not tagged as a reviewer. Kobe's checklist does not include a step to verify domain rule compatibility or to tag Bird on architecture decisions that touch regulated workflows. At the time, this was not negligence — the review process had simply never been designed with a regulatory compliance gate. **~2025-10 (architecture goes live):** Order state transitions begin flowing through the event-sourced pipeline. Projection lag is typically under 200ms. No audit trail completeness monitoring is deployed. Pippen's operational posture treats 200ms lag as a non-issue for functional purposes, which it is — for non-regulatory queries. **2026-01-19 (incident):** FCA conducts an audit. Regulators query state records within seconds of transitions occurring. In 4 cases, the asynchronous projection has not yet completed, and the query returns stale state. The FCA flags six weeks of records as non-compliant with COBS 11.2A. The team learns of the compliance gap for the first time — from the regulator, not from internal systems. ## 3. Contributing Factors Each of the following factors was independently necessary for this outcome. Removing any one of them would likely have prevented or detected the issue before the FCA did. **Factor 1 — Terminology mismatch between domain and architecture language.** Bird's domain rule used the phrase "immediately durable." MJ read "durable" as a storage property (data persisted to disk). Bird meant it as a regulatory property (committed to the compliance store synchronously, queryable immediately). Both interpretations are valid in their respective contexts. The term was never explicitly disambiguated because no process required it. **Factor 2 — Bird was not included in ADR-014 review.** The domain rule predated the architecture decision. Bird owned the rule and could have caught the mismatch instantly. But the ADR review process did not include a step to identify and tag the domain authority for affected business domains. Bird was not tagged, and therefore did not participate. **Factor 3 — Kobe's review checklist had no regulatory compliance gate.** Kobe's review was thorough for its scope — but that scope was architectural soundness, not domain rule compatibility. The checklist was designed for engineering quality, not regulatory alignment. This was a reasonable design for a checklist that predated the team's work on regulated instruments. **Factor 4 — No audit trail completeness monitoring.** Pippen's operational setup did not include alerting for projection lag on compliance-critical read models. The 200ms lag was operationally invisible because it caused no user-facing errors. Without monitoring that specifically tracked "time from event write to compliance-queryable projection," there was no way to detect the gap before a regulator did. **Factor 5 — Implicit assumption that all read model consumers had the same latency tolerance.** The architecture treated all projection consumers uniformly. No distinction was made between "dashboard showing order status to an internal user" (tolerates seconds of lag) and "compliance record store queryable by regulators" (tolerates zero lag). This assumption was never stated, so it was never challenged. ## 4. Learnings **We now know that "durable" means different things in engineering and regulatory contexts**, which means any architecture decision involving regulated data must explicitly define durability in regulatory terms, not just infrastructure terms. Glossary alignment between domain and architecture is not optional — it is a compliance control. **We now know that ADR review without domain authority sign-off can silently introduce regulatory violations**, which means the review process itself is a compliance-relevant control and must include a mandatory step to identify whether the decision touches a regulated domain. **We now know that projection lag on compliance-critical read models is a regulatory risk, not just a performance metric**, which means audit trail completeness must be monitored with alerting thresholds calibrated to regulatory query windows, not to user experience. **We now know that the absence of an incident is not evidence of compliance**, which means we cannot rely on "no one complained" as a signal that regulated workflows are functioning correctly. Compliance requires proactive verification. **We now know that review checklists are silent about what they don't cover**, which means the checklist itself must be periodically audited against the team's current regulatory obligations. A checklist that was complete last quarter may have gaps this quarter. ## 5. Action Items - `[PREVENT]` **Add "Regulated Domain Impact" gate to ADR review process.** Every ADR must include a section: "Does this decision affect data flows for regulated instruments?" If yes, Bird (or the designated domain authority) must be tagged as a required reviewer. **Owner: Kobe. Due: 2026-02-07.** - `[PREVENT]` **Create a shared glossary of regulatory terms with engineering definitions.** Starting with "immediately durable," "queryable," "audit trail," and "compliance record store." Each term gets a domain definition (Bird), an engineering implementation requirement (MJ), and a test assertion (Kobe). **Owner: Magic (coordination), Bird (domain definitions), MJ (engineering mapping). Due: 2026-02-14.** - `[PREVENT]` **Implement synchronous write-through to compliance record store for regulated instrument state transitions.** The event log remains the source of truth for replay, but a synchronous write to the compliance store must occur before the transition is confirmed to the client. This is a targeted fix — not all projections need to be synchronous, only compliance-critical ones. **Owner: MJ (architecture), Shaq (implementation). Due: 2026-02-21.** - `[DETECT]` **Deploy audit trail completeness monitoring.** An alerting rule that fires when any regulated instrument state transition has not appeared in the compliance-queryable store within N milliseconds (threshold to be agreed with Bird based on COBS 11.2A requirements). **Owner: Pippen. Due: 2026-02-14.** - `[DETECT]` **Implement a nightly reconciliation job.** Compares event log entries for regulated instruments against the compliance record store. Any discrepancy generates a P1 alert. **Owner: Pippen (operational setup), Shaq (implementation). Due: 2026-02-21.** - `[MITIGATE]` **Backfill the 6-week compliance gap.** Replay events from the event log into the compliance record store for the affected period. Validate completeness against the event log. Produce an evidence package for the FCA demonstrating remediation. **Owner: MJ (replay design), Shaq (execution), Bird (FCA evidence review). Due: 2026-02-07.** - `[PROCESS]` **Audit Kobe's review checklist against current regulatory obligations quarterly.** Each quarter, Bird provides the current set of regulatory constraints. Kobe verifies the checklist covers them. Gaps are addressed before the next ADR cycle. **Owner: Kobe (checklist maintenance), Bird (regulatory input). First review: 2026-02-14, then quarterly.** ## 6.
