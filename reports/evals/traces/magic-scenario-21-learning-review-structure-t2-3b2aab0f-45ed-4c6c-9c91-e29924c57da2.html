# Learning Review: Seasonal Pricing Cache Staleness Incident (2026-03-12) ## Situation Snapshot On 2026-03-12, a cache invalidation refactor (`pricing-service v2.4.1`, PR #1847) was deployed to the product pricing service. The change altered how TTL values are computed for category-specific cache entries, inadvertently bypassing the seasonal category's domain-mandated 15-minute TTL cap during active promotional windows. For 20 minutes (14:22–14:42 UTC), 847 customer transactions on seasonal items were charged previous-day promotional prices. The deployment was rolled back after Pippen's price divergence alert detected the anomaly. Financial impact: incorrect pricing on 847 orders; customer trust impact under assessment. ## Timeline 1. **Pre-deployment (days prior):** The cache invalidation ticket was scoped as a "performance optimization." Because it was not categorized as a pricing rule change, Bird (domain authority) was not consulted during spec or implementation. No seasonal-specific acceptance criteria were written. 2. **Implementation:** Shaq refactored TTL computation to pass TTL directly from the call site rather than reading it from `CategoryConfig`. This was a reasonable simplification for the standard product path, which has no special TTL constraints. 3. **Code Review:** Kobe reviewed PR #1847, verified unit tests passed, and confirmed the standard path logic was correct. The seasonal category's TTL override in `CategoryConfig` was not checked against the new call-site pattern. At the time, this was a reasonable scope for the review given the ticket's "performance optimization" framing. 4. **Staging Validation:** Shaq tested in staging against the "standard" category. Seasonal products were not present in the staging fixture set, so the broken path was never exercised. This was not known to be a gap because nobody had flagged seasonal TTL as in-scope. 5. **14:22 UTC — Deployment:** `pricing-service v2.4.1` goes live. The seasonal category begins serving cache entries with an incorrect (longer) TTL, retaining stale promotional prices. 6. **14:22–14:38 UTC (16 min silent window):** Customers purchasing seasonal items see yesterday's promotional prices. No alert fires because the price divergence has not yet crossed the 5% threshold for enough items. 7. **14:38 UTC — Alert fires:** Pippen's price divergence anomaly alert triggers, comparing pricing-service responses against the source-of-truth price feed. 8. **14:38–14:42 UTC (4 min):** Team investigates the alert, correlates it with the recent deployment, and executes rollback. 9. **14:42 UTC — Rollback complete.** Correct prices restored. ## Contributing Factors Each of these factors was independently necessary; none alone was sufficient to cause the incident. 1. **Ticket scoping excluded domain consultation.** The cache invalidation work was framed as "performance optimization," which placed it outside the workflow where Bird reviews domain rule implications. The seasonal TTL rule — a business-critical constraint — was invisible to both implementer and reviewer because neither had reason to look for it. 2. **Domain rule encoded only in configuration, not in tests.** The 15-minute TTL cap for seasonal products during promotional windows exists in `CategoryConfig` but has no corresponding automated test asserting that cache TTL respects it. The rule was a silent invariant rather than an enforced contract. 3. **Staging fixtures did not cover seasonal products.** The staging environment lacked seasonal category data, so the broken code path was never exercised before production. At the time, this was not recognized as a coverage gap because the ticket appeared category-agnostic. 4. **Code review focused on changed code, not downstream invariants.** Kobe's review correctly verified the new logic for the standard path. The review did not extend to checking whether all `CategoryConfig` overrides were still honored, because the PR's framing suggested no category-specific behavior was being altered. This is a normal and rational review scope — the gap is systemic, not individual. 5. **Alert threshold introduced a 16-minute detection delay.** The 5% price divergence threshold is tuned to avoid false positives, which means real divergences take time to accumulate enough signal. This is an intentional tradeoff, but it meant nearly the entire incident duration occurred before detection. ## Learnings 1. **We now know that "performance optimization" tickets can silently alter domain-critical behavior**, which means any change touching category-specific logic — even if framed as non-functional — must be flagged for domain review. The category of a ticket is not a reliable proxy for its blast radius. 2. **We now know that the seasonal TTL rule is a silent invariant with no automated enforcement**, which means any future cache or TTL refactor is equally likely to violate it. Domain rules that exist only in configuration without corresponding test coverage are hidden landmines. 3. **We now know that staging fixture coverage has a seasonal-shaped hole**, which means any code path specific to the seasonal category has been shipping to production without pre-prod validation. This likely extends to other underrepresented categories. 4. **We now know that 16 minutes is the practical detection floor for price divergence at current thresholds**, which means for time-sensitive pricing categories, the current alerting posture accepts a window of exposure that may be too wide. The 4-minute response time after detection was healthy. 5. **We now know that the boundary between "how we cache" and "what business rules constrain caching" is not visible in our ticket taxonomy**, which means engineers and reviewers lack a reliable signal for when domain authority input is needed. ## Action Items - `[PREVENT]` **Add contract tests for category-specific TTL rules.** Create automated tests that assert each category's cache TTL respects its `CategoryConfig` constraints, including the seasonal 15-minute cap during promotional windows. **Owner: Shaq. Due: 2026-03-31.** - `[PREVENT]` **Add seasonal products to staging fixtures.** Ensure the staging environment includes representative data for all product categories with custom domain rules, starting with seasonal. **Owner: Pippen. Due: 2026-04-04.** - `[DETECT]` **Evaluate a category-aware alerting tier.** Investigate whether the price divergence alert can use tighter thresholds (or faster sampling) for categories with time-sensitive pricing rules, without increasing false positive rate unacceptably. **Owner: Pippen. Due: 2026-04-11.** - `[PROCESS]` **Introduce a "domain-touch" label in the ticket workflow.** When a ticket modifies behavior that is parameterized by category, region, or other domain-partitioned config, it must be labeled for Bird's review regardless of its primary classification. Define the trigger criteria collaboratively with Bird. **Owner: Bird + Coach K. Due: 2026-04-07.** - `[MITIGATE]` **Add a pre-deploy smoke check for seasonal pricing consistency.** As part of the deployment pipeline, run a lightweight comparison of seasonal product cache TTLs against `CategoryConfig` expected values before traffic is shifted. **Owner: Shaq. Due: 2026-04-14.** ## Preserving the Learning - **ADR: "Domain rules must have corresponding contract tests"** — Document the principle that any business rule encoded in configuration (TTL caps, pricing windows, category-specific behavior) must have a test that breaks if the rule is violated. This ADR should reference this incident as the motivating example. **Owner: Bird.** - **Checklist addition: PR review checklist gains a "CategoryConfig impact" line item** — For any PR touching cache, TTL, or category-parameterized logic, the reviewer must verify that `CategoryConfig` overrides are still honored. Added to the team's review template. **Owner: Kobe.** - **Staging fixture coverage policy** — Document a policy that staging fixtures must include at least one representative entity for every category with custom domain rules, maintained as new categories are added. **Owner: Pippen.** - **Ticket taxonomy guide update** — Add examples to the team's ticket scoping guide showing when a "performance optimization" ticket crosses into domain-rule territory, using this incident as the canonical case. **Owner: Coach K.**
