# Learning Review: user_preferences Migration Deadlock and Recovery **Date of incident**: 2026-02-14 **Review produced**: 2026-03-24 --- ## Situation Snapshot During a scheduled maintenance window on 2026-02-14, a database migration adding a nullable JSON column to the `user_preferences` table (800k rows) was applied to production. The accompanying backfill job deadlocked with a live API transaction after completing roughly 60% of rows. A disagreement between team members on the recovery path (rollback vs. forward-fix) was resolved in favor of rollback, which then required replaying 4 hours of accumulated preference update events from the event log. The preference API served stale, pre-migration data for 47 minutes while the replay executed. No data was permanently lost. --- ## Timeline 1. **Maintenance window opens.** The migration is applied per the existing runbook. A nullable JSON column is added to `user_preferences`. A backfill job — a single, unbatched `UPDATE` across all 800k rows — begins executing. 2. **~11 minutes in: Deadlock detected.** The backfill's table-wide `UPDATE` conflicts with a live API transaction performing a `SELECT FOR UPDATE` on the same table. The backfill fails. At this point, ~60% of rows have been backfilled; ~40% have `NULL` in the new column. The team knows the table is in a partially migrated state but the API's nullable fallback means the service is not hard-down. 3. **Disagreement on recovery path.** Kobe advocates immediate rollback to a known-good state, reasoning that the deadlock's root cause is not yet fully understood and a partially migrated table is an unacceptable risk. MJ advocates completing the backfill (forward-fix), reasoning that the already-migrated 60% of rows are consistent, that rollback introduces a novel and unrehearsed event-replay operation, and that the event log is accumulating live events whose replay correctness is itself a risk. Both parties note there is no pre-agreed decision framework for this situation. The decision is made under time pressure. 4. **Rollback chosen.** The team rolls back the migration. This reverts the column addition and the 60% of backfilled rows. However, preference update events that accumulated during the migration window (roughly 4 hours of events) now need to be replayed to restore the pre-migration state to currency. 5. **Event replay designed and executed.** 4 hours are spent designing the replay procedure (it had not been pre-planned). The replay itself executes in 47 minutes. During this combined window, the preference API serves stale data. 6. **Service restored.** The replay completes successfully. No data is lost. The migration is not reattempted during this window. --- ## Contributing Factors Each of the following factors independently contributed to the incident's scope and duration. No single factor is "the root cause." **1. Unbatched backfill on a live table.** The backfill was a single `UPDATE` statement against 800k rows with no batching. This maximized lock duration and lock scope, creating a wide window for deadlock with any concurrent transaction touching the same table. The runbook did not specify a batch size threshold, and the implementer (Shaq) reasonably followed the runbook as written. **2. Live API traffic during the migration window.** The maintenance window did not isolate the `user_preferences` table from live API writes. A `SELECT FOR UPDATE` from a live API transaction was the direct deadlock partner. Whether this traffic was expected during the window, and whether the window's definition included write quiescence on the target table, was not specified. **3. Semantic ambiguity in the nullable column.** Bird's domain rule — preference data must be consistent per user row, with no partial states — was not reconciled with the migration's design choice to add a *nullable* column. A `NULL` in the new column is technically valid at the schema level but semantically ambiguous: it could mean "user has not set a preference" or "user's row has not yet been migrated." This ambiguity was not surfaced or resolved before the migration was approved. It meant the team could not confidently assess whether the partially migrated state was safe to operate in, which directly fueled the rollback-vs-forward-fix disagreement. **4. No pre-agreed decision framework for partial migration failure.** Neither rollback nor forward-fix had been designated as the default recovery path for a migration that fails mid-backfill. Both Kobe and MJ were reasoning from first principles under pressure. The absence of a shared framework meant the team spent decision-making time during the incident that could have been spent on execution. **5. Event replay was an unrehearsed operation.** The rollback path required replaying 4 hours of accumulated events — a procedure that had not been pre-designed or tested. The 4-hour design time for the replay was a direct consequence of this. Ironically, this was one of MJ's arguments *against* rollback, and the actual recovery experience validated the concern: the replay was successful, but it was novel, unplanned, and time-consuming. **6. No SLA or target for migration recovery time.** There was no defined acceptable duration for degraded service during migration recovery. This made it difficult to evaluate whether 47 minutes of stale data was "acceptable" or a severity escalation, both during and after the incident. --- ## The Kobe/MJ Disagreement: A Signal, Not a Verdict Kobe and MJ each held a defensible position. Kobe prioritized returning to a *known-good state* and avoiding the unknown risk of further deadlocks. MJ prioritized *minimizing novel operations* and avoiding the replay-correctness risk that rollback introduced. Both positions were internally consistent. Both were grounded in real risks. MJ's model treated it as safe *if the migrated rows are individually consistent*. Bird's domain rule (no partial states per user row) could support either interpretation depending on whether "partial" refers to individual row consistency or table-wide completeness. The team had not aligned on which definition of "partial" governs migration decisions. 2. **What the risk hierarchy is for recovery operations.** Kobe weighted the risk of an *unknown future deadlock* higher than the risk of an *unrehearsed replay*. MJ weighted the opposite. Neither weighting is objectively correct — it depends on organizational context, risk tolerance, and the specific failure mode. The team had no shared framework to resolve this. The fact that both engineers independently identified the missing framework ("we had no pre-agreed criteria") is itself a valuable signal. It was a **failure of preparation** — the team had not done the pre-work to make this decision automatic. --- ## Learnings - **We now know that unbatched backfills on tables with live write traffic create deadlock risk that scales with table size**, which means any migration runbook for tables above a size threshold must specify batching strategy and batch size, and must document whether live traffic is expected during the window. - **We now know that a nullable column can be schema-valid but domain-ambiguous**, which means migration design reviews must include a domain-level assessment (not just a schema-level assessment) of what `NULL` means in the new column — specifically, whether it is a valid business state or a migration artifact. - **We now know that rollback and forward-fix are both complex operations with distinct risk profiles**, which means the team needs a pre-agreed decision framework — not to eliminate judgment, but to give the on-call team a shared starting point rather than requiring first-principles reasoning under pressure. - **We now know that event replay is a non-trivial recovery operation that requires pre-design**, which means any migration whose rollback path depends on event replay must have that replay procedure documented, tested, and time-estimated *before* the migration is approved. - **We now know that the Kobe/MJ disagreement pattern — two experienced engineers reaching opposite conclusions from the same facts — is a reliable signal that a shared mental model is missing**, which means future disagreements of this shape should be treated as an immediate prompt to identify the missing shared definition, not as a tiebreaker decision. --- ## Action Items - `[PREVENT]` **Add batching requirements to the migration runbook.** Define a row-count threshold (e.g., >10k rows) above which backfills must be batched.
