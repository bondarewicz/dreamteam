Let me design a real-time courier bidding system architecture. I have clear constraints here - let me classify and proceed. Classification: `none` — sufficient constraints exist. I have NFRs (500 concurrent bid windows, 15s timeout, atomic acceptance, 30 couriers/zone, real-time), scale requirements, and clear business rules. Let me think through this carefully. Key technical challenges: 1. Atomic "first to accept wins" — classic race condition problem 2. 15-second timeout with auto-assignment fallback 3. Real-time push to ~30 couriers per job 4. 500 simultaneous bid windows The core insight is that "first to accept wins" is fundamentally a distributed lock problem, and Redis handles this perfectly with atomic operations and low latency. I'm thinking through the architecture as a set of interconnected services: an Order Service that kicks off the process, a Bid Window Manager handling the 15-second lifecycle, a Bid Processor that atomically handles acceptance attempts, a Courier Gateway managing WebSocket connections to mobile apps, an Assignment Service that finalizes once the bid resolves, and a Courier Eligibility Service determining which couriers can bid. The flow moves from order placement through event publishing, then the Bid Window Manager queries eligible couriers and sets up a Redis entry with a 15-second TTL. Couriers get notified via WebSocket, and when someone taps accept, the Bid Processor runs an atomic Redis operation to claim the job—if successful, the winner gets notified and losers see the job is taken; if the window expires, the highest-rated courier gets auto-assigned. Writing the atomic acceptance logic... For timeouts, I can either use Redis key expiry with keyspace notifications or set up a delay queue with a sorted set that workers poll. The WebSocket layer keeps persistent connections mapped to courier IDs and zones, so when a bid window opens, I fan out to all eligible couriers in that zone. At scale, with 500 concurrent bid windows and roughly 30 couriers per zone, that's around 15,000 concurrent connections handling acceptance attempts over 15 seconds—Redis handles this load easily.
