The backfill failed at 60% completion due to a deadlock with a live API transaction. The team disagreed on the recovery path — rollback vs. forward-fix — and ultimately rolled back, requiring a 4-hour event log replay. The preference API served stale (pre-migration) data for 47 minutes while the replay executed. No data was permanently lost. --- ## Timeline 1. **Maintenance window opens.** Migration applied: new nullable JSON column added to `user_preferences`. Backfill begins as a single, unbatched `UPDATE` across all 800k rows. This approach followed the runbook, which did not specify a batch size threshold. 2. **~11 minutes in.** A deadlock occurs between the backfill `UPDATE` and a live API transaction performing `SELECT FOR UPDATE` on the same table. The backfill fails. At this moment, 60% of rows have the new column populated; 40% remain `NULL`. 3. **Disagreement surfaces.** Kobe advocates immediate rollback to a known-good state, citing the principle that a partially-migrated table is an unknown state and the deadlock root cause is unconfirmed. MJ advocates forward-fix to complete the remaining 40%, reasoning that rollback introduces a novel, unrehearsed operation (event log replay) and that the already-migrated rows are consistent. Both recognize there is no pre-agreed decision framework for this scenario. 4. **Decision made under pressure: rollback.** The team chooses rollback. The rollback itself reverts the column, but now 4 hours of accumulated preference update events need to be replayed to restore the pre-migration state to current. 5. **Event log replay designed and executed.** Pippen reports the replay design took significant coordination time. The replay itself ran for 47 minutes. During this window, the preference API served pre-migration data — semantically correct but stale. 6. **Service restored.** Replay completes successfully. No data lost. Post-incident review begins. --- ## Contributing Factors Each of the following factors independently contributed to the incident's severity. No single factor is the "root cause." 1. **Unbatched backfill on a live-traffic table.** The backfill was a single `UPDATE` across 800k rows. This maximized lock duration and therefore maximized the window in which a deadlock with a concurrent transaction could occur. The runbook did not specify batching, and Shaq followed the runbook as written. The omission in the runbook — not Shaq's execution — is the contributing factor. 2. **Live API traffic during the maintenance window.** The `SELECT FOR UPDATE` that triggered the deadlock came from a live API transaction. Either the maintenance window did not fully quiesce write traffic to the table, or the team's definition of "maintenance window" did not include pausing API transactions that acquire row-level locks. This boundary was not explicitly defined. 3. **Semantic ambiguity in the nullable column.** Bird's domain rule — preference data must be consistent per user row — was not reconciled with the migration design's use of a nullable column. A `NULL` in the new column could mean "user has no preference" or "user hasn't been migrated yet." This ambiguity was not surfaced during migration approval. Had it been, the team might have designed differently (e.g., a sentinel value for "not yet migrated," or a feature flag to gate reads until backfill completes). 4. **No pre-agreed rollback vs. forward-fix decision framework.** When the backfill failed, the team had no shared criteria for choosing between rollback and forward-fix. Both Kobe and MJ were reasoning from first principles under time pressure. This is not a failure of either individual's judgment — it is a gap in the team's shared operational playbook. 5. **No rehearsed recovery path for partial migration failure.** The event log replay that followed rollback was designed ad hoc during the incident. While it succeeded, it introduced 47 minutes of degraded service and required significant coordination time. A pre-rehearsed recovery procedure would have reduced both duration and stress. 6. **No SLA or target for migration recovery time.** Pippen noted the absence of an SLA for migration recovery. Without a target, the team had no way to evaluate whether 47 minutes of degradation was acceptable or unacceptable — making it harder to prioritize investment in faster recovery mechanisms. --- ## The Kobe/MJ Disagreement: A Signal, Not a Verdict This section intentionally does not resolve which position was correct. The disagreement itself is the most valuable signal from this incident. **Kobe's reasoning** prioritized returning to a known-good state. A partially-migrated table is an unknown state; the deadlock's root cause was unconfirmed; the API's nullable fallback meant rollback was safe at the schema level. This reasoning is conservative and grounded in the principle that uncertainty should be reduced, not compounded. **MJ's reasoning** prioritized minimizing novel operations under pressure. The forward-fix would complete an operation already 60% done. Rollback would introduce a new operation — event log replay — that had never been rehearsed. The replay carried its own correctness risk. This reasoning is grounded in the principle that an unrehearsed recovery is itself a risk. **Both positions were internally consistent and well-reasoned.** The disagreement arose because the team lacked: - **A shared definition of "known-good state."** Kobe treated it as "the schema before migration." MJ treated it as "the data state before migration" — and argued that restoring data state via replay was itself an unknown operation. - **A shared risk model for partial migration.** The team had not pre-classified which risks they considered more dangerous: a partially-migrated table (Kobe's concern) or an unrehearsed replay operation (MJ's concern). - **A pre-agreed decision authority for live incidents.** Under pressure, the team debated the merits. A pre-agreed framework (e.g., "if backfill is <75% complete, rollback; if >=75%, forward-fix" or "the on-call DBA makes the final call within 5 minutes") would have shortened the decision window regardless of which path was chosen. **The lesson is not "Kobe was right" or "MJ was right."** The lesson is that the team's shared understanding did not extend to this failure mode, and that gap became visible only under pressure. Closing the gap means building the shared framework *before* the next incident, not adjudicating this one after the fact. --- ## Learnings 1. **We now know that our runbook does not specify batching thresholds for backfill operations**, which means any future migration against a table with concurrent write traffic carries the same deadlock risk. Bird's domain rule about per-row consistency was not part of the migration review process. 4. **We now know that the team has no shared decision framework for rollback vs. forward-fix**, which means any future partial failure will produce the same ad hoc debate under pressure. The Kobe/MJ disagreement was a symptom of this gap, not a cause of the incident's duration. 5. **We now know that event log replay, while successful, took 47 minutes and was designed ad hoc**, which means our actual migration recovery time is measured in hours (design + execution), not minutes.
