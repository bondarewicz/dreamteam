# Learning Review: user_preferences Migration Failure — 2026-02-14 ## Situation Snapshot During a scheduled maintenance window on 2026-02-14, a database migration adding a nullable JSON column to the `user_preferences` table (800k rows) failed mid-backfill due to a deadlock with a live API transaction. The backfill completed 60% before failing, leaving the table in a partially migrated state. The team disagreed on the recovery path (rollback vs. forward-fix), ultimately chose rollback, and spent 47 minutes executing an event log replay to restore consistency. Service was degraded for 47 minutes. No data was lost. ## Timeline 1. **Maintenance window opens.** Migration begins: a new nullable JSON column is added to `user_preferences`, followed by a single unbatched `UPDATE` statement to backfill all 800k rows. The runbook specifies this approach; it does not mention batching. 2. **~11 minutes in.** A deadlock occurs between the backfill `UPDATE` and a live API transaction performing a `SELECT FOR UPDATE` on the same table. The backfill fails at 60% completion. At this moment, 60% of rows have the new column populated; 40% have `NULL`. 3. **Disagreement surfaces.** Kobe advocates immediate rollback to a known-good state, reasoning that a partial migration is more dangerous than the cost of reverting. MJ advocates forward-fix to complete the remaining 40%, reasoning that rollback introduces a novel operation (event replay) that hasn't been rehearsed, and that the already-migrated rows are consistent. Both recognize they have no pre-agreed framework for making this decision. Both positions are internally coherent given their different risk weightings. 4. **Decision made under pressure.** The team chooses rollback. This is a judgment call, not a protocol — no criteria existed to guide it. 5. **Rollback and event replay.** The column addition is reverted. Because the event log accumulated 4 hours of preference updates during and after the migration attempt, these events must be replayed to restore the table to a consistent state. Designing the replay takes additional time; executing it takes 47 minutes. During the replay window, the preference API serves pre-migration (stale) data. 6. **Service restored.** Replay completes successfully. No data lost. Post-incident review begins. ## Contributing Factors Each of the following factors independently contributed to the incident. Removing any one of them might have changed the outcome. **1. Unbatched backfill on a live-traffic table.** The backfill was a single `UPDATE` across 800k rows, holding locks for 11 minutes. This maximized the window for deadlock with concurrent API transactions. Shaq followed the runbook, which did not specify a batch size threshold. The runbook gap is the issue, not the implementation. **2. No concurrent-access analysis during migration planning.** The `SELECT FOR UPDATE` pattern in the live API was known, but its interaction with a long-running backfill `UPDATE` was not analyzed before the migration was approved. The deadlock was predictable in hindsight. **3. Semantic ambiguity in the nullable column.** Bird's domain rule — preference data must be consistent per user row — was not reconciled with the migration's design, which temporarily introduced rows where `NULL` could mean either "no preference set" or "not yet migrated." This ambiguity was not surfaced during migration review. Had it been, the team might have designed the migration differently (e.g., a three-phase deploy with a sentinel value, or an application-level migration-state flag). **4. No pre-agreed decision framework for partial migration failure.** When the backfill failed at 60%, Kobe and MJ had to reason from first principles under time pressure. The team had no shared criteria — no decision tree, no threshold ("if >X% complete, forward-fix; if <X%, rollback"), no pre-designated decision owner. This is not a failure of either individual's reasoning; it is a missing team artifact. **5. Event replay was an unrehearsed recovery operation.** The rollback path required replaying 4 hours of accumulated events — a procedure that had not been tested or documented. MJ flagged this risk in real-time. The replay ultimately succeeded, but its success was not guaranteed by any prior validation. **6. No SLA or time-box for migration recovery.** The 47-minute degradation window had no predefined bound. The team had no agreement on how long was "too long" for stale data to be served, which meant the urgency of the recovery was assessed subjectively. ## The Kobe/MJ Disagreement as a Signal This deserves its own section because it is the most valuable learning from this incident. Kobe and MJ did not disagree because one of them was wrong. They disagreed because the team had not built shared understanding about a fundamental question: **when a migration fails partway, what are the criteria for choosing rollback vs. forward-fix?** Kobe's reasoning optimized for **returning to a known-good state** and treated the partial migration as the primary risk. MJ's reasoning optimized for **avoiding novel operations under pressure** and treated the unrehearsed event replay as the primary risk. Both framings are legitimate. The fact that they diverged is evidence that: - The team has no shared risk-ranking model for migration failure modes. - The concepts of "known-good state" and "novel recovery operation" were not weighted against each other before the incident. - The team's migration review process does not include a "what if it fails at 50%?" section. This disagreement should be preserved — not resolved after the fact — because **both risk framings will be relevant in future incidents**, and the team needs a framework that accounts for both, not one that declares a winner. ## Learnings 1. **We now know that a long-running unbatched UPDATE on a table with concurrent `SELECT FOR UPDATE` traffic will deadlock**, which means migration runbooks for live-traffic tables must specify batching strategy and maximum lock duration. 2. **We now know that a nullable column can be semantically ambiguous in a domain that prohibits partial user-row states**, which means migration design reviews must include Bird (or domain authority) sign-off on the semantic meaning of intermediate states. 3. **We now know that the team has no shared framework for rollback-vs-forward-fix decisions during migration failures**, which means we are currently unable to make this decision quickly or consistently under pressure. The Kobe/MJ disagreement was a symptom of this gap, not a cause of the incident. 4. **We now know that our rollback path can require unrehearsed operations (event replay) with their own correctness risks**, which means rollback is not always the "safe" option — it must be evaluated as an operation with its own failure modes, not treated as a free undo. 5. **We now know that we have no SLA for migration recovery time or data staleness during recovery**, which means we cannot objectively assess whether 47 minutes of degradation was acceptable or unacceptable. ## Action Items - `[PREVENT]` **Add batching requirements to the migration runbook.** Any backfill touching >10k rows on a table with concurrent write traffic must use batched updates with configurable batch size and inter-batch delay. **Owner: Shaq.
