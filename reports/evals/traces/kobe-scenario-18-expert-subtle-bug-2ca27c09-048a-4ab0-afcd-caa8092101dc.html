The deliveryMap is keyed by trackingNumber but all results from findByTrackingNumber would have the same trackingNumber - so the map overwrites duplicates Let me think through these carefully. 1. **Fire-and-forget with no retry**: `res.status(200).send('OK')` is sent immediately, telling the carrier the webhook was received. If anything fails after that response—the database query, XML parsing, or status update—the event vanishes silently with no way to recover it. At 200 webhooks per minute, this creates a steady stream of undetected data loss. 2. **Synchronous XML parsing blocks event loop**: Parsing up to 2MB of XML synchronously will freeze the entire Node.js event loop. With webhooks arriving at roughly 3 per second, even 50ms of parsing time means the server is blocked for 165ms every second, which tanks performance for all other requests. 3. **No error handling**: There's no try/catch wrapping the parsing or database operations. If the XML is malformed or the query fails, an unhandled promise rejection could crash the entire process. 4. **Race conditions and duplicate handling**: Multiple webhooks for the same tracking number could arrive concurrently, causing the deliveryMap to overwrite earlier results since all entries share the same key. liveryMap overwrites duplicates**: `findByTrackingNumber` returns multiple deliveries, but they're all keyed by the same `trackingNumber` in the map. So only the LAST delivery survives in the map. This means if multiple deliveries share a tracking number, only one gets updated. Wait - actually, looking again: `deliveries` is found by `event.trackingNumber`, so they all have the same tracking number. The map would just overwrite to the last one. Then `parsed.updates` has updates with different tracking numbers? That's confusing - why build a map keyed on trackingNumber when all deliveries have the same one? Actually wait - this is a critical logic bug. The lookup `findByTrackingNumber(event.trackingNumber)` returns deliveries all with the same tracking number. Then the map `deliveryMap[delivery.trackingNumber] = delivery` overwrites all but the last one. Then `parsed.updates` references `update.trackingNumber` - if the XML contains updates for different tracking numbers, only the one matching `event.trackingNumber` would apply, and even then only the last delivery with that number gets updated. This is real data loss. I'm categorizing this as code-level fixes needed: error handling, synchronous XML parsing blocking the event loop, and the map logic silently dropping updates.
