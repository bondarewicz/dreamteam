# Learning Review: user_preferences Migration Deadlock and Recovery **Date of incident**: 2026-02-14 **Review author**: Magic Johnson (Context Synthesizer) **Review date**: 2026-03-24 --- ## 1. Situation Snapshot During a scheduled maintenance window on 2026-02-14, a database migration added a nullable JSON column to the `user_preferences` table (800k rows) and initiated a backfill from an event log. The backfill failed at 60% completion due to a deadlock with a live API transaction. The team disagreed on the recovery path — Kobe advocated rollback, MJ advocated forward-fix — and ultimately rolled back. The rollback required replaying 4 hours of accumulated preference updates from an event log, resulting in 47 minutes of degraded service. No data was lost, but the incident exposed several gaps in migration planning, runbook detail, and team decision-making frameworks. --- ## 2. Timeline **Pre-incident**: The migration was approved and scheduled for the maintenance window. The runbook specified the migration steps but did not include a batch size for the backfill, a deadlock mitigation strategy, or criteria for rollback vs. forward-fix in the event of partial failure. The semantic ambiguity of a nullable column (no preference vs. not-yet-migrated) was not surfaced during review. **T+0**: Maintenance window opens. Shaq executes the migration per the runbook: schema change (add nullable JSON column) followed by a single-statement backfill UPDATE across all 800k rows. **T+11m**: The backfill deadlocks with a live API transaction performing a `SELECT FOR UPDATE` on the same table. The backfill fails at approximately 60% completion. The table is now in a partially-migrated state — 60% of rows have the new column populated, 40% do not. **T+11m to ~T+25m**: The team assesses the situation. Two recovery positions emerge: - **Kobe's position**: Roll back immediately. The partially-migrated state is unknown territory. The API can tolerate a missing column via nullable fallback. A known-good state is preferable. The deadlock root cause is not yet confirmed resolved, making a forward-fix risky. - **MJ's position**: Fix forward by completing the backfill. The 60% already migrated are consistent. Rollback introduces a new, unrehearsed operation (event replay of 4 hours of accumulated events) with its own correctness risks. The forward path completes a known operation; the backward path starts a new one. Both Kobe and MJ recognized in real time that they had no pre-agreed decision framework for this scenario. They were reasoning from first principles under time pressure. **Decision point**: The team chose rollback. At the time, this decision made sense: the team had higher confidence in restoring a known-good state than in completing an operation that had just failed due to an incompletely understood deadlock. The forward-fix required confidence that the deadlock would not recur, and that confidence did not exist. **T+25m to ~T+4h25m**: Rollback is executed. The schema change is reverted. The team then designs an event log replay to recover the 4 hours of preference updates that had accumulated during and after the migration attempt. This replay design took approximately 4 hours — it was a novel operation that had not been rehearsed. **T+4h25m to ~T+5h12m**: The event replay executes successfully over 47 minutes. Service is fully restored. During the gap between rollback and replay completion, the preference API served pre-migration data. --- ## 3. Contributing Factors Each of the following factors independently contributed to the incident's scope and duration. None is "the root cause" — the incident required all of them to converge. **Unbatched backfill on a live table.** The backfill was a single UPDATE statement against 800k rows. This maximized the lock duration and therefore the deadlock window. Shaq followed the runbook, which did not specify batching. The runbook's silence on batch size was interpreted as "not needed," which was a reasonable interpretation given the information available. **Concurrent write traffic during the maintenance window.** A live API transaction was performing `SELECT FOR UPDATE` against the same table during the backfill. The maintenance window did not include a write-traffic freeze on the affected table, and the runbook did not identify this as a risk. The team's mental model of "maintenance window" did not include "exclusive access to the migration target." **Semantic ambiguity in the nullable column.** Bird's domain rule is that preference data must be consistent per user — partial states are not permitted at the row level. However, a nullable column cannot distinguish between "user has no preference" and "user's preference hasn't been migrated yet." This ambiguity was not surfaced during migration review. Had it been, the team might have chosen a different migration strategy (e.g., a separate migration-status flag, or a two-phase migration with a feature flag). **No pre-agreed rollback/forward-fix decision criteria.** When the failure occurred, the team had no shared framework for deciding between rollback and forward-fix. This meant the decision had to be made under pressure using ad hoc reasoning. Both Kobe's and MJ's positions were internally consistent and well-reasoned — the disagreement was not about competence but about the absence of shared criteria. (See Section 4 for more on this.) **Event replay was an unrehearsed recovery path.** The rollback path required replaying 4 hours of events from a log. This operation had not been designed, tested, or documented before the incident. Its 4-hour design time dominated the recovery timeline. Regardless of which recovery path the team chose, the lack of a rehearsed recovery plan for partial migration failure would have introduced delay. **No SLA for migration recovery time.** The team had no defined acceptable recovery window, which meant the 47-minute degradation could not be evaluated against a standard. This also meant the pre-migration risk assessment had no target to measure against. --- ## 4. Learnings **On the Kobe/MJ disagreement — a signal, not a verdict.** The disagreement between Kobe and MJ was not a failure of either person's judgment. It was a signal that the team's shared understanding was incomplete in a specific and important way: *the team had never articulated criteria for when to roll forward versus roll back during a partial migration failure.* Kobe reasoned from a safety-first heuristic: a known-good state is preferable to an unknown state when the failure mode is not fully understood. This is sound reasoning — it prioritizes predictability and minimizes exposure to uncharacterized risk. MJ reasoned from a total-cost-of-recovery heuristic: the forward path completes a known operation, while the backward path introduces a new, unrehearsed operation (event replay) with its own risk profile. This is also sound reasoning — it accounts for the fact that rollback is not free and may introduce risks that are harder to see because they are novel. Both positions were correct given their premises. The disagreement existed because the team had not, prior to the incident, agreed on which premises take priority in this class of situation. Under pressure, each team member defaulted to their own reasoning framework, which is exactly what happens when shared frameworks are absent. **We now know that** the absence of a pre-agreed rollback/forward-fix framework will surface as a real-time disagreement during the incident, which is the worst possible time to resolve it. This means such frameworks must be established during migration planning, not during migration execution. **We now know that** a single-statement backfill on a table with concurrent write traffic creates a deadlock risk proportional to the number of rows and the duration of the lock. This means backfill operations should be batched by default, with the batch size calibrated to the table's write traffic profile. **We now know that** a nullable column is not semantically equivalent to "no migration needed." A nullable column in a partially-migrated table creates an ambiguity that violates domain rules even if it satisfies schema rules. This means migration plans must include Bird's domain review of the intermediate states, not just the final state. **We now know that** the event replay recovery path, while ultimately successful, took 4 hours to design because it was novel. This means any migration plan that lists "replay from event log" as a rollback strategy must include a pre-tested replay procedure, or it is not actually a rollback plan — it is a hope. **We now know that** our concept of "maintenance window" does not include exclusive write access to the migration target. This means either our maintenance windows need to be redefined, or our migrations need to be designed to tolerate concurrent writes. --- ## 5.
