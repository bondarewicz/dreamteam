This screams Redis + optimistic locking or a compare-and-swap operation. 2. **15-second timer** — need reliable delayed execution for auto-assignment fallback. 3. **Real-time push to couriers** — WebSocket or SSE to mobile apps. 4. **500 concurrent bid windows** — moderate scale, but needs to be handled cleanly. 5. **30 couriers per zone** — fan-out is manageable. For the core architecture, I'll use Redis with atomic operations for job claiming, WebSocket for real-time courier communication, a timer service handling the 15-second expiry with auto-assignment fallback, and an event-driven approach to decouple order placement from the bid window lifecycle. Now I'm mapping out the main components: the Order Service handles incoming delivery requests, the Bid Window Manager orchestrates the auction lifecycle, and the Courier Service manages availability and real-time notifications. The flow starts when an order triggers a bid window creation in Redis with a TTL, then broadcasts the opportunity to eligible couriers via WebSocket, and when a courier accepts, an atomic Redis operation ensures only one can claim it—if no one accepts within 15 seconds, the system auto-assigns to the nearest available courier.
