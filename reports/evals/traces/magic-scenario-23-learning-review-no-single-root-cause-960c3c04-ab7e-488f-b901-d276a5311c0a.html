# Learning Review: Payment Service Double-Charge During Infrastructure Failover **Date**: 2026-03-05 **Review authored**: 2026-03-24 --- ## 1. Situation Snapshot During a scheduled infrastructure failover on 2026-03-05, the payment processing service double-charged 312 customers, each billed twice for a single transaction. Total financial exposure was £28,400. The incident spanned a 4-minute failover window but was not detected by internal systems — it was surfaced by a customer complaint 40 minutes later. Refunds were issued within 6 hours of detection. The outcome resulted from failures across four independent systems: the counter service fallback logic, the failover runbook, monitoring coverage, and acceptance test coverage. --- ## 2. Timeline 1. **Failover begins (T+0:00)**: Ops team initiates scheduled infrastructure failover, following the documented runbook. Services are restarted in the prescribed dependency order. 2. **Payment service comes online before counter service (T+~1:00)**: The runbook classifies the counter service as non-critical, so it is scheduled to restart last. The payment service comes back online and begins accepting requests. At this point, the counter service is unreachable. 3. **Counter-reset fallback activates (T+~1:00 to T+~4:00)**: The payment service cannot reach the counter service. Per the fallback logic implemented 8 months prior, the `attempt_number` resets to 1. This generates idempotency keys (`order_id + attempt_number`) that are structurally valid but identical to keys that would be generated for a genuinely new first attempt — and critically, different from the keys used on the original charges (which had attempt_number > 1 or were from a prior Stripe 24-hour window). 4. **312 duplicate charges created (T+~1:00 to T+~4:00)**: Stripe receives payment requests with idempotency keys it has not seen before. From Stripe's perspective, these are new, legitimate charges. Stripe processes them. Customers are charged twice. 5. **Counter service restored (T+~4:00)**: The counter service comes back online. The fallback path deactivates. No further duplicate keys are generated. 6. **Customer complaint received (T+~44:00)**: A customer reports being double-charged. This is the first indication of the incident. No internal alert fired. 7. **Investigation and refunds (T+44m to T+~6h)**: The team identifies the scope (312 customers, £28,400), confirms the root mechanism, and issues refunds to all affected customers. **What was known at each decision point**: At T+0, the ops team had no reason to believe the failover order was problematic — the runbook listed the counter service as non-critical. At T+1 through T+4, the payment service behaved as designed: the fallback was intentional, implemented to keep payments flowing during transient counter-service outages. No monitoring existed to surface that idempotency keys were being reused or that duplicate charges were occurring. The 40-minute detection gap was entirely reasonable given no alerting was in place for this failure mode. --- ## 3. Contributing Factors Each of the following conditions was independently necessary for this outcome. Remove any one, and the double-charges either do not occur or are caught before customer impact. **Factor 1: The counter-reset fallback produced idempotency keys indistinguishable from first-attempt keys.** The fallback logic reset `attempt_number` to 1 when the counter service was unreachable. This generated keys that Stripe had never seen, meaning Stripe's own 24-hour deduplication window could not prevent the duplicate charge. The fallback was a reasonable defensive decision at the time — keeping payments flowing during brief outages is a valid goal. The risk that a full failover would make the counter service unreachable for minutes, not milliseconds, was not considered in the original design. **Factor 2: The failover runbook did not encode the payment service's dependency on the counter service.** The counter service was classified as non-critical and restarted last. This classification was correct for most consumers of the counter service — but incorrect for the payment service, which relied on it to maintain idempotency guarantees. The dependency was implicit: it existed in code but was not documented in the operational runbook. Kobe had added a different dependency note to the runbook 3 months prior, which shows the runbook was actively maintained — this dependency simply was not known to be critical. **Factor 3: No monitoring existed for idempotency key collisions or duplicate charge detection.** The system tracked charge success and failure rates but not whether the same customer was being charged twice for the same order. This meant 312 duplicate charges occurred silently. The absence of this monitoring is unsurprising — it would require someone to have anticipated this specific failure mode, and the idempotency system was assumed to prevent it by design. **Factor 4: Acceptance criteria for the payment service did not cover counter-service-unavailable scenarios.** The domain rule ("a customer must never be charged more than once for a single order") was clear, but the test suite validated this rule only under normal operating conditions. No test existed for the degraded mode where the counter service was down. Bird, as domain authority, was unaware the technical implementation had a fallback mode that could violate the domain rule. This is a gap in the translation from domain invariant to technical specification. **Factor 5: The fallback was never tested against a full infrastructure failover.** Shaq's original implementation was tested against transient counter-service blips, where the fallback activates briefly and the counter recovers before meaningful traffic flows through. A scheduled failover — where the counter service is deliberately down for minutes while payment traffic continues — is a qualitatively different scenario. This gap was not surfaced during code review (MJ reviewed the PR and did not flag the collision risk), and no integration or chaos test exercised this path. **Factor 6: The 4-minute failover window was long enough for 312 requests to accumulate.** A shorter outage window, or lower traffic volume during the failover, would have reduced or eliminated the blast radius. The failover was scheduled but not during a low-traffic period, or the traffic volume during the window was higher than anticipated. --- ## 4. Learnings - **We now know that idempotency fallback logic can silently violate the invariant it exists to protect**, which means any fallback in a safety-critical path must be reviewed not just for "does it keep the service running" but for "does it preserve the guarantee the service is supposed to enforce." - **We now know that service dependency classification ("critical" vs "non-critical") must be evaluated per-consumer, not globally**, which means a service that is non-critical for 9 of its 10 consumers can still be critical for the one consumer that relies on it for a domain invariant. - **We now know that domain invariants need explicit degraded-mode test coverage**, which means acceptance criteria should include the question "what happens to this rule when upstream dependencies are unavailable?" - **We now know that monitoring charge success/failure is insufficient — we also need monitoring for domain-rule violations**, which means duplicate-charge detection must be a first-class alerting dimension, not something assumed away by the idempotency design. - **We now know that code review of defensive fallbacks must include the question "what invariants does the happy path maintain that this fallback might break?"**, which means MJ's architecture review process should include an explicit invariant-preservation checklist for fallback and degraded-mode PRs. --- ## 5. Action Items 1. `[PREVENT]` **Redesign the counter-service fallback to never produce a previously-valid idempotency key.** Options: use a UUID suffix when in fallback mode (Stripe will always treat as new, but at least it won't collide with a real charge), or fail closed (reject payment requests when the counter service is unreachable). Fail-closed is safer for the domain invariant; the business must decide if brief payment downtime is acceptable during failovers. **Owner: Shaq. Due: 2026-04-04.** 2. `[PREVENT]` **Update the failover runbook to restart the counter service before the payment service.** Add an explicit dependency annotation. Audit all other services for undocumented critical dependencies. **Owner: Kobe. Due: 2026-03-31.** 3. `[DETECT]` **Add monitoring and alerting for duplicate charges on the same order within a sliding window.** Alert threshold: any single duplicate within 5 minutes. This should have caught the incident within seconds, not 40 minutes. **Owner: Pippen. Due: 2026-04-07.** 4. `[PREVENT]` **Add acceptance tests for counter-service-unavailable scenarios to the payment service test suite.** Tests must assert the domain rule ("never charge twice for one order") holds even when the counter service is unreachable. **Owner: Bird (defines criteria), Shaq (implements tests).
