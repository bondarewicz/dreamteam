A financial services platform needs to track every change to a loan application's status and data for regulatory audit purposes. The audit trail must be immutable and must capture who changed what and when. Currently they use a single "loan_applications" table with update-in-place semantics; they add a "last_updated_by" column but lose historical state. The compliance team requires: full history of all field changes, ability to reconstruct the application state at any point in time, and immutable audit log. The engineering team is considering event sourcing. Evaluate event sourcing for this use case. Use your full output schema.

Let me analyze this financial services loan application audit trail use case and evaluate event sourcing for it. This is a well-constrained problem. Let me classify it first: - We have clear NFRs: immutable audit trail, full history, point-in-time reconstruction, regulatory compliance - We have a clear domain: loan applications - We have a clear current state: update-in-place with lost history - The question is whether event sourcing is the right pattern Classification: `none` — sufficient constraints exist to make architecture decisions. Let me think through this carefully. Event sourcing is actually a very strong fit here because: - It naturally provides immutable audit trails and full history - Point-in-time reconstruction works by replaying events to any timestamp - The append-only log structure is inherently tamper-evident for compliance I should weigh this against alternatives like CDC with audit tables or temporal SQL tables, but event sourcing has a clear advantage: domain events capture the intent behind state changes, not just the diffs. The trade-offs favor ES here—it's purpose-built for exactly this problem. The downsides are real though: CQRS becomes almost necessary for querying, the team needs to learn new patterns, event schema evolution gets tricky, and reporting requires more thought. A simpler audit log table would be easier to implement and maintain, but it doesn't give you the same natural reconstruction capabilities or intent preservation that event sourcing provides. For financial services with regulatory requirements, the alignment is too good to ignore—ES handles the core compliance needs natively, even if the implementation complexity is higher.
