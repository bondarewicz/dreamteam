# Learning Review: Invoicing Service Supplier ID Mismatch Incident **Date of review**: 2026-03-24 **Incident window**: 2026-02-28 to 2026-03-03 (3 days silent failure) **Prepared by**: Magic Johnson (Context Synthesizer) --- ## 1. Situation Snapshot The Invoicing Service ingested supplier invoices via an external webhook, persisting `supplier_id` as a raw string. When a new sub-provider began sending float-formatted IDs (e.g., `"12345.0"` instead of `"12345"`), records were written successfully but silently failed to reconcile downstream. The mismatch went undetected for 3 days, producing a £47,000 discrepancy in the monthly reconciliation report. The issue was caught by the finance team during month-end close — not by any automated system. --- ## 2. Contributing Factors Each of the following factors independently contributed to the outcome. No single factor is "the root cause" — the incident required all of them to occur together. 1. **Upstream format change without notification.** The webhook provider onboarded a new sub-provider whose system generates float-formatted IDs. This change was not announced to the consuming team. The team's risk acceptance was based on the provider's stated behaviour, which was accurate at the time but became stale. 2. **Accepted risk that outlived its context.** TD-291 was deprioritised based on a reasonable assessment: the provider confirmed integer-format IDs, and the timeline was tight. However, the risk acceptance was not revisited when the next sprint began, and TD-291 remained in the backlog without a forcing function to re-evaluate it. 3. **No input validation or normalisation at the ingestion boundary.** The system persisted `supplier_id` exactly as received. This is a defensible design choice for an opaque identifier — but it meant the system had zero tolerance for upstream format drift, creating a brittle coupling to the provider's undocumented formatting behaviour. 4. **Silent failure in the reconciliation job.** The downstream job logged warnings for unmatched `supplier_id` values but did not alert. The warning volume was low enough to fall below the manual review threshold. The job was designed to handle known-good data, not to act as a data quality gate. 5. **No contract or schema enforcement on the webhook input.** There was no validation layer that would reject or flag structurally unexpected `supplier_id` formats (e.g., values containing a decimal point). The webhook accepted any string, making upstream changes invisible. 6. **Detection gap: 3-day window with no automated catch.** The only detection mechanism was month-end financial reconciliation — a human, batch process. There was no near-real-time check that could surface a growing population of unmatched records. --- ## 4. Learnings - **We now know that upstream provider guarantees about data format are point-in-time statements, not durable contracts.** This means any risk acceptance based on "the provider currently sends X" must include a mechanism to detect when that assumption becomes false — the assumption itself has a shelf life. - **We now know that exact-string matching on externally-sourced identifiers is a brittle reconciliation strategy.** This means any identifier that crosses a system boundary should either be normalised at ingestion or matched with tolerance downstream. Preferably both. - **We now know that WARNING-level logs without alerting thresholds create a dead zone where problems accumulate invisibly.** This means warnings in reconciliation or data-quality jobs need either an absolute-count alert (e.g., "any new unmatched supplier_id") or a trend-based alert, not just a volume threshold. - **We now know that deprioritised risk items need a re-evaluation trigger, not just a backlog position.** This means tech debt items that represent accepted risks should have an explicit "revisit by" date or a condition that forces re-triage — sitting in a backlog is not the same as being managed. - **We now know that the boundary between "the system worked as designed" and "the system failed" can be a single upstream change we have no visibility into.** This means our ingestion boundaries need defensive validation even when the current provider behaviour is well-understood. --- ## 5.
