Use PostgreSQL Advisory Locks for Nightly Batch Coordination **Status:** Accepted **Date:** 2026-03-24 **Deciders:** MJ (architecture), team review **Technical area:** Distributed coordination, batch scheduling ## Context The nightly reconciliation job has historically run on a single application node with no coordination mechanism. The system is being expanded to a multi-node deployment. Without coordination, every node would execute the batch job independently, causing duplicate processing and potential data inconsistency. We need a distributed mutual-exclusion mechanism that ensures exactly one node runs the reconciliation job per night. ## Decision Use PostgreSQL session-level advisory locks (`pg_try_advisory_lock`) to coordinate nightly batch execution across nodes. Each node, on its scheduled trigger, will attempt to acquire a well-known advisory lock ID using the non-blocking `pg_try_advisory_lock`. The node that acquires the lock proceeds with the batch job; all other nodes log a skip and exit. The lock is released when the session ends (either via explicit unlock or connection close). ## Rationale - **No new infrastructure.** PostgreSQL is already in the stack. Advisory locks are a built-in primitive with no additional deployment, monitoring, or failover requirements. - **Sufficient for the workload.** The job runs once per night with a single coordinator. There is no need for high-throughput lock contention handling, TTL-based expiration, or complex leader election. - **Crash-safe by default.** If the lock-holding node crashes, PostgreSQL detects the dropped connection and releases the session-level lock automatically. The next scheduled trigger can acquire it without manual intervention. ## Alternatives Considered | Alternative | Why rejected | |---|---| | **Redis SETNX with TTL** | Introduces a new infrastructure dependency (Redis) solely for one nightly lock. Operationally disproportionate. | | **ZooKeeper** | Heavyweight coordination service; unjustified for a single low-frequency batch job. | | **Custom heartbeat table in PostgreSQL** | Requires application-level TTL logic, cleanup of stale rows, and polling. Advisory locks provide the same semantics with less code and fewer failure modes. | ## Trade-offs and Risks | Risk | Severity | Mitigation | |---|---|---| | Advisory locks are invisible outside the database (no dashboard, no CLI to list them) | Low | Log lock acquisition/release events. Query `pg_locks` for diagnostics when needed. | | No built-in TTL -- a zombie connection holding the lock blocks all other nodes | Low | Session-level locks auto-release on connection drop. Set `statement_timeout` and `idle_in_transaction_session_timeout` as a safety net. | | Connection poolers in transaction mode (e.g. PgBouncer) silently break session-level locks | Medium | Ensure the batch job uses a dedicated, non-pooled connection or a pooler in session mode. Document this requirement. | | No proof that the job actually completed successfully (only that the lock was acquired) | Medium | Add post-job completion logging and a monitoring alert if no successful completion is recorded by a deadline (e.g. 06:00). | ## Consequences - All application nodes must connect to the same PostgreSQL primary for lock coordination. - A single, well-documented lock ID constant must be maintained in the codebase. - Operators should monitor for successful batch completion, not just lock acquisition. - If the system later requires more sophisticated coordination (multiple jobs, sub-second leader election, cross-service locks), this decision should be revisited in favor of a dedicated coordination service.
