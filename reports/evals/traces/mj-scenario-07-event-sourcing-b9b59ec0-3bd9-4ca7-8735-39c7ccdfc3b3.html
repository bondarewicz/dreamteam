The audit trail must be immutable and must capture who changed what and when. The compliance team requires: full history of all field changes, ability to reconstruct the application state at any point in time, and immutable audit log. The engineering team is considering event sourcing. Evaluate event sourcing for this use case. Use your full output schema.

Current state: update-in-place losing history Event sourcing evaluation: - ES stores every state change as an immutable event — natural fit for audit - But ES brings significant complexity: eventual consistency, event schema evolution, rebuilding projections, etc. - Alternative: Event-carried state transfer with an append-only audit log table alongside the current state table (simpler) - Another alternative: CDC (Change Data Capture) from the existing table For a loan application, the domain complexity is moderate with a well-defined lifecycle (submitted → under review → approved/denied → funded → closed), so the number of state transitions is manageable. Now I'm weighing whether full event sourcing makes sense here or if something simpler would work better. Full ES gives perfect audit trails and point-in-time reconstruction, but it's complex—projections, event versioning, eventual consistency between models. A dual-write approach with a current state table plus an append-only audit log is simpler to reason about and keeps existing queries unchanged, though there's a consistency risk if the two diverge. For a regulated financial domain where audit trails are mandatory, event sourcing actually becomes compelling because the audit trail itself becomes the source of truth.
