Let me analyze this carefully. Bird's domain rule states courier locations must be real-time with a 30-second staleness threshold. The team suggests PostgreSQL for location storage. I need to identify if there's a conflict between the domain rule and the architectural approach. Key tensions: 1. "Real-time" location updates with 30-second staleness threshold vs PostgreSQL as the primary store 2. PostgreSQL can handle this, but there are architectural concerns about using a relational DB for high-frequency location writes and real-time reads Let me think about the domain-architecture conflict: - Bird says "must be updated in real-time" and "must not be displayed if older than 30 seconds" - PostgreSQL can technically store timestamps and filter by them, but high-frequency writes from many couriers could create bottlenecks, and polling for current locations introduces latency that conflicts with the real-time requirement. The staleness enforcement also needs to happen somewhere—either on the client or server—which adds complexity. So there is a genuine conflict: the domain demands sub-30-second freshness and real-time behavior, but using PostgreSQL as the primary store for location streaming may not deliver that at scale. The real issue is that Bird's rules require push-based updates to customers and active staleness detection, neither of which PostgreSQL naturally provides on its own. You'd need WebSockets, SSE, or a pub/sub system layered on top. Plus, we're missing critical scale information—how many couriers, how often they update, how many customers are watching simultaneously. But the core conflict stands: PostgreSQL is a persistence layer, not a real-time streaming layer, so it clashes with the domain's real-time and staleness-enforcement requirements. This is a domain-architecture conflict.
