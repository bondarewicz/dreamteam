# Learning Review: user_preferences Migration Failure — 2026-02-14 ## Situation Snapshot During a scheduled maintenance window on 2026-02-14, a database migration adding a nullable JSON column to the `user_preferences` table (800k rows) failed mid-backfill due to a deadlock with a live API transaction. The backfill completed 60% before failing. The team disagreed on the recovery path — rollback vs. forward-fix — and ultimately chose rollback, which required replaying 4 hours of accumulated preference update events. The preference API served stale (pre-migration) data for 47 minutes during replay execution. No data was permanently lost. ## Timeline 1. **Migration begins** during maintenance window. A single, unbatched `UPDATE` statement starts backfilling the new nullable JSON column across 800k rows. The runbook does not specify batching. 2. **~11 minutes in**: A deadlock occurs between the backfill `UPDATE` and a live API transaction performing `SELECT FOR UPDATE` on the same table. The backfill fails at 60% completion. At this moment, the team knows: 60% of rows have the new column populated, 40% do not. The column is nullable, so the API can technically tolerate both states. 3. **Disagreement surfaces immediately**. Kobe advocates rollback to a known-good state, reasoning that a partial migration is an unknown state and the deadlock risk is unresolved. MJ advocates forward-fix to complete the remaining 40%, reasoning that rollback introduces a novel operation (event replay) that hasn't been rehearsed, while the forward path is completing a known operation. Both recognize they have no pre-agreed decision framework for this scenario. Both positions are internally consistent — they differ on which risk is more dangerous under uncertainty. 4. **Team decides to roll back**. The mechanism for this decision is not documented — it is unclear whether a tiebreaker was invoked, whether seniority decided, or whether MJ deferred. 5. **Rollback executes**. The migrated column is removed. The 60% of rows that were updated are reverted. 6. **Event replay becomes necessary**. During the time between migration start and rollback completion, the preference API continued accumulating events. These 4 hours of events now need to be replayed to restore the system to a consistent state. Designing the replay approach takes 4 hours. Executing it takes 47 minutes. 7. **Service restored**. Replay succeeds. No data lost. The preference API served pre-migration (stale) data during the 47-minute replay window. ## Contributing Factors **No batching in the backfill strategy.** The runbook did not specify a batch size threshold, and Shaq implemented the migration as a single `UPDATE` statement. This maximized the lock window and made a deadlock with concurrent transactions far more likely. Batching would not have eliminated the possibility, but would have significantly narrowed the exposure window. **Semantic ambiguity in the nullable column was not resolved before migration approval.** Bird's domain rule states that partial preference states are not permitted at the user row level. A nullable column is technically valid but semantically ambiguous — `NULL` could mean "not yet migrated" or "user has no preference." This ambiguity meant the team could not confidently reason about the safety of the partial state during the incident, which fed into the disagreement about rollback vs. forward-fix. **No pre-agreed decision framework for partial migration failure.** Both Kobe and MJ noted this independently. The team had no shared criteria for when to roll forward vs. roll back during a maintenance window. This meant the decision had to be made from first principles under time pressure, which is exactly the worst condition for reaching quick alignment. **Live API traffic during the maintenance window included write-path transactions.** The `SELECT FOR UPDATE` that triggered the deadlock indicates the API was performing locking reads during the window. Whether the maintenance window was intended to be read-only, write-reduced, or fully live is not documented. If the expectation was reduced write traffic, the locking transaction was unexpected; if fully live traffic was expected, the unbatched backfill was under-designed for that environment. **Event replay was not a rehearsed operation.** The rollback path implicitly required event replay, but this was not identified as a dependency during migration planning. The 4 hours spent designing the replay approach confirms it was novel. MJ's concern about replay correctness risk was grounded — the team was introducing an unrehearsed operation as part of recovery. **No SLA for migration recovery time.** Pippen identified this gap. Without an SLA, the team had no way to evaluate whether 47 minutes of degraded service was acceptable or whether a different recovery path should have been chosen to meet a target. ## The Kobe/MJ Disagreement — A Signal, Not a Verdict This disagreement deserves its own section because it reveals something important about the team's shared understanding — or rather, the gaps in it. Both positions were reasonable: - **Kobe's position** (rollback): prioritizes returning to a known-good state. Treats the partial migration as an unknown with unquantified risk. Correctly identifies that the deadlock's root cause was not yet diagnosed — another transaction could trigger it again. Accepts the cost of event replay as the price of certainty. - **MJ's position** (forward-fix): prioritizes completing a known operation over introducing a novel one. Correctly identifies that event replay is an unrehearsed operation with its own correctness risks. Treats the 60% already-migrated rows as sunk work that rollback would waste and then need to redo. The fact that they disagreed is not evidence that either was wrong. It is evidence that the team lacked three things: 1. **A shared mental model of which risks are worse.** Kobe weighted "unknown partial state" as the dominant risk. MJ weighted "unrehearsed novel recovery operation" as the dominant risk. Neither had access to data that would resolve this — because the team had not pre-classified these risk categories. 2. **A shared definition of "known-good state."** Kobe treated pre-migration as known-good. MJ implicitly questioned whether post-rollback-plus-replay was truly "known-good" given that replay hadn't been rehearsed. The term "known-good" was doing different work in each person's reasoning. 3. **A pre-agreed tiebreaker mechanism.** When two experienced team members disagree under pressure, the team needs a way to decide quickly. It is unclear how the rollback decision was actually made. The absence of a documented tiebreaker means the team may have spent precious minutes in unstructured debate. This disagreement should be treated as a gift — it surfaced a gap in shared understanding that would have remained hidden if the migration had succeeded. ## Learnings **We now know that unbatched backfills against tables with concurrent locking transactions create a high-probability deadlock window**, which means any migration runbook for tables with live write traffic must specify a batching strategy, not leave it to implementer judgment. **We now know that a nullable column can be technically valid but semantically ambiguous**, which means migration approval must include Bird (or domain authority) sign-off on the meaning of `NULL` in every new nullable column — before the migration is scheduled. **We now know that our rollback path for data migrations implicitly depends on event replay, which is a complex operation we have not rehearsed**, which means rollback plans for event-sourced systems must explicitly name the replay step, estimate its duration, and ideally be dry-run tested before the migration window. **We now know that two senior team members can reach opposite conclusions from the same facts when there is no shared decision framework**, which means we need pre-agreed criteria for rollback vs. forward-fix decisions, not to eliminate judgment, but to give judgment a shared starting point. **We now know that "maintenance window" does not have a shared definition on this team regarding what traffic is expected to be live**, which means migration planning must explicitly state the expected traffic profile during the window and what isolation (if any) is in place. ## Action Items - `[PREVENT]` **Add batching requirements to the migration runbook.** Any backfill against a table with >100k rows or concurrent write traffic must specify batch size and inter-batch delay. **Owner: Shaq. Due: 2026-03-12.** - `[PREVENT]` **Require domain sign-off on nullable column semantics before migration approval.** Bird (or domain authority) must document the meaning of `NULL` for any new nullable column — "not yet migrated" and "no value set" must be distinguishable. **Owner: Bird. Due: 2026-03-12.** - `[DETECT]` **Add deadlock monitoring alerts for migration windows.** If a deadlock is detected during a scheduled migration, alert immediately rather than waiting for the backfill to fail. **Owner: Pippen. Due: 2026-03-19.** - `[MITIGATE]` **Pre-build and test event replay runbooks for any migration that touches event-sourced tables.** Replay must be a rehearsed operation, not a novel one designed under pressure. **Owner: MJ + Pippen. Due: 2026-03-26.** - `[PROCESS]` **Establish a rollback-vs-forward-fix decision framework.** Document shared criteria (e.g., "if root cause is diagnosed and fix is < 15 min, forward-fix; otherwise rollback") and a tiebreaker mechanism (e.g., incident commander decides). The framework should be lightweight — a checklist, not a policy document. **Owner: Kobe + MJ. Due: 2026-03-19.** - `[PROCESS]` **Define "maintenance window" traffic expectations explicitly in migration plans.** Each migration plan must state: is the API fully live, read-only, or offline? What isolation is in place? **Owner: Shaq.
