# Learning Review: Payment Double-Charge During Infrastructure Failover **Date of Incident**: 2026-03-05 **Review Date**: 2026-03-24 **Prepared by**: Magic Johnson (Context Synthesizer) --- ## 1. Situation Snapshot During a scheduled infrastructure failover on 2026-03-05, the payment processing service double-charged 312 customers, each billed twice for a single transaction. Total financial exposure was £28,400. The incident spanned a 4-minute failover window and was detected 40 minutes later via customer complaint — not by internal monitoring. Refunds were issued within 6 hours. The root mechanism was an idempotency key collision caused by the interaction of four independent system behaviors: a counter-reset fallback, Stripe's 24-hour deduplication window, a runbook that misclassified a dependency, and the absence of duplicate-charge monitoring. --- ## 2. Timeline | Time | Event | What was known | Decision / Action | |------|-------|---------------|-------------------| | T+0 min | Scheduled infrastructure failover begins | Ops team executing a planned, routine failover per the runbook | Proceed with failover per documented runbook | | T+0–1 min | Services begin restarting in dependency order per runbook | Counter service classified as non-critical; restarted last | Payment service brought online before counter service — this was the documented procedure | | T+1–4 min | Payment service online, counter service unreachable | Payment service logs (if any) would show fallback activation, but no one was watching for this specifically | 312 payment requests arrive; the counter-reset fallback sets `attempt_number = 1`, generating idempotency keys identical in structure to the original charges but treated as novel by Stripe | | T+4 min | Counter service comes back online | Failover complete; all services healthy from an infrastructure perspective | Ops marks failover as successful | | T+44 min | Customer complaint received about a double charge | First external signal that something went wrong | Investigation begins | | ~T+60 min | Engineering identifies the counter-reset fallback as the cause | Team realizes the reset produced colliding idempotency keys that Stripe treated as new charges | Decision to issue bulk refunds | | T+6 hours | All 312 affected customers refunded | Full scope identified via payment logs | Incident closed operationally; review initiated | --- ## 3. Contributing Factors Each of the following conditions was independently necessary for this outcome. Remove any single one and the double charges do not occur. **Factor 1: The counter-reset fallback defaulted to `attempt_number = 1` when the counter service was unreachable.** Shaq implemented this 8 months prior as a defensive measure to keep the payment service functional during transient counter-service outages. The intent — graceful degradation — was sound. The consequence — generating idempotency keys that could collide with previously-used keys — was not analyzed at the time. The fallback was designed for brief network blips, not for a scenario where the counter service would be deliberately down for minutes during a full failover. **Factor 2: The failover runbook classified the counter service as non-critical and restarted it last.** The runbook's dependency ordering did not capture the payment service's reliance on the counter service for idempotency correctness. Kobe had previously added dependency notes for other services but this particular dependency was not documented. From the ops team's perspective, the restart order was correct per documented policy. **Factor 3: Stripe's idempotency deduplication operates on a 24-hour window with exact key matching.** Stripe will reject a duplicate charge only if the exact same idempotency key is submitted within 24 hours. Because the counter reset produced `attempt_number = 1` for retry attempts, and the original successful charges had used different attempt numbers, the keys were technically distinct. Stripe processed them as legitimate new charges. The external payment processor behaved exactly as designed — it had no way to know these were duplicates. **Factor 4: No monitoring existed for idempotency key uniqueness or duplicate charge detection.** Pippen's notes confirm that operational monitoring tracked charge success/failure rates but not whether the same order was being charged multiple times. This meant the 312 double charges produced no alerts. The incident was silent from an observability standpoint. **Factor 5: Acceptance criteria for the payment service did not include counter-service-unavailable scenarios.** Bird's domain rule — "a customer must never be charged more than once for a single order" — was clear. But the test suite validated this rule only under normal operating conditions. No test exercised the fallback path. The gap between the domain invariant and its technical enforcement was invisible because the failure mode had never been specified as a test case. **Factor 6: The architecture review of the fallback PR did not flag the key-collision risk.** MJ reviewed Shaq's original fallback PR and assessed it as a reasonable defensive measure. The review did not trace the downstream effect of a counter reset on idempotency key uniqueness within Stripe's deduplication window. This was not negligence — the interaction between the fallback behavior, the key format, and Stripe's deduplication semantics is a second-order effect that is easy to miss in a code review focused on the immediate change. **Factor 7: 312 payment requests arrived during the 4-minute failover window.** Had the failover occurred during a low-traffic period, fewer or no customers would have been affected. The traffic volume during the window determined the blast radius, not the existence of the bug. --- ## 4. Learnings - **We now know that graceful degradation in payment-adjacent services can violate payment invariants**, which means fallback modes for any service involved in charge processing must be evaluated against the domain rule "no customer is charged twice," not just against "service stays available." - **We now know that idempotency key construction has implicit dependencies on external state (the counter service)**, which means the idempotency mechanism is only as reliable as the least-available component in the key's derivation chain. Idempotency keys should be derivable from immutable inputs wherever possible. - **We now know that our runbook's dependency model is incomplete**, which means any service that contributes to a financial correctness invariant must be classified as critical for restart ordering, regardless of whether it is required for basic service health. - **We now know that monitoring charge success rates is not the same as monitoring charge correctness**, which means we need a distinct observability signal for "same order charged more than once" that is independent of whether each individual charge succeeded. - **We now know that domain invariants need explicit failure-mode test cases**, which means acceptance criteria should include not just "does the happy path preserve the invariant?" but "does each known degraded mode preserve the invariant?" - **We now know that architecture reviews of fallback mechanisms need to trace key-collision and state-reset side effects end-to-end**, which means PR reviews for fallback logic in payment paths should include a specific checklist item: "Can this fallback produce inputs that an external system (e.g., Stripe) will treat as distinct when they should be treated as identical?" --- ## 5. Action Items 1. **`[PREVENT]`** — Redesign the idempotency key to be derived solely from immutable values (e.g., `order_id` + a hash of the original charge request) so that counter-service availability cannot affect key uniqueness. **Owner**: Shaq. **Due**: 2026-04-04. 2. **`[PREVENT]`** — Remove the counter-reset fallback. If the counter service is unreachable, the payment service should fail the request and return a retryable error rather than silently degrade idempotency. **Owner**: Shaq. **Due**: 2026-04-04 (same PR as item 1). 3. **`[PREVENT]`** — Update the failover runbook to classify the counter service as critical and restart it before the payment service. Audit all other services for undocumented dependencies on financial-correctness invariants. **Owner**: Kobe. **Due**: 2026-04-07. 4. **`[DETECT]`** — Add monitoring and alerting for duplicate charges: trigger an alert when the same `order_id` appears in more than one successful charge within a 24-hour window. **Owner**: Pippen. **Due**: 2026-04-11. 5. **`[DETECT]`** — Add a metric tracking idempotency key fallback activation. Any activation of the counter-reset fallback (until it is removed) should fire an immediate alert. **Owner**: Pippen. **Due**: 2026-03-31 (fast-track, before the redesign ships). 6. **`[MITIGATE]`** — Implement an automated duplicate-charge reconciliation job that runs hourly, compares charges against orders, and auto-queues refunds for duplicates. This reduces blast radius even if detection fails. **Owner**: Shaq + Pippen. **Due**: 2026-04-18. 7. **`[PROCESS]`** — Add acceptance test cases for counter-service-unavailable scenarios to the payment service test suite. More broadly, establish a pattern: every domain invariant with "must never" semantics requires at least one test exercising each known degraded mode. **Owner**: Bird (criteria) + Shaq (implementation). **Due**: 2026-04-11. 8. **`[PROCESS]`** — Add a PR review checklist item for payment-path changes: "If this introduces a fallback or degraded mode, trace the effect on idempotency keys and external processor deduplication." **Owner**: MJ. **Due**: 2026-03-31. --- ## 6. Preserving the Learning - **ADR: "Idempotency Keys Must Be Derived From Immutable Inputs"** — Document the decision that idempotency keys in the payment domain must never depend on mutable or availability-contingent state
