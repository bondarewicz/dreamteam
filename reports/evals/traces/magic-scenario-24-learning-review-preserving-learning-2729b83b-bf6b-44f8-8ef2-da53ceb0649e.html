# Learning Review: FCA Audit Trail Non-Compliance Incident (2026-01-19) ## 1. Situation Snapshot During a routine FCA audit on 2026-01-19, regulators discovered that 6 weeks of order state transition records for regulated instruments were non-compliant with COBS 11.2A audit trail requirements. The root issue was an architectural mismatch: MJ's event-sourced order state system projected transitions asynchronously (~200ms lag), while Bird's pre-existing domain rule required synchronous, immediately-durable commits to the compliance record store before confirming transitions to clients. In 4 specific cases, regulators queried state within seconds of a transition and received stale data. The gap affected all regulated instrument order transitions over a 6-week window, exposing the organization to regulatory sanction. ## 2. Timeline **~2025-09-19 (4 months before incident):** MJ designed order state transitions using event sourcing. ADR-014 was written and reviewed. At this point, MJ understood "durable" in the infrastructure sense — fsync'd to disk — and the event log satisfied that property. The asynchronous projection to the read model was a deliberate architectural choice for decoupling and replay capability. This made sense given the information MJ had: the event log *was* durable, and 200ms projection lag seemed negligible for all known consumers. **~2025-09-19 (same window):** Kobe reviewed ADR-014 against his standard checklist: scalability, failure modes, operational complexity, security. The review passed. Kobe's checklist did not include regulatory compliance or domain rule compatibility checks. This was not an oversight by Kobe — it was a gap in the review process itself. No one flagged Bird as a required reviewer. **~2025-09-19 (same window):** Bird was not tagged for review of ADR-014. Bird's domain rule — that each state transition must be committed to the compliance record store *before* the transition is confirmed to the client — was documented in the domain model under "Regulated Instruments — Audit Requirements." This rule predated ADR-014. The rule and the ADR existed in separate review streams with no cross-reference mechanism. **2025-09 through 2026-01 (~4 months):** The system operated normally. The 200ms projection lag caused no visible issues. No audit trail completeness monitoring existed. No one had reason to question the architecture — it worked correctly for all non-regulatory use cases. **2026-01-19:** FCA conducted an audit. Regulators queried state records within seconds of transitions occurring. In 4 cases, the asynchronous projection had not completed, returning stale state. The FCA flagged 6 weeks of records as non-compliant with COBS 11.2A. The team learned about the problem from the FCA, not from internal monitoring. ## 3. Contributing Factors Each of the following factors was independently necessary for the incident to occur in this form. Removing any single one would have either prevented or detected the problem earlier. **Terminology divergence between architecture and domain.** MJ and Bird both used the word "durable" but meant different things. MJ meant database-level durability (fsync). Bird meant regulatory-grade durability: synchronous commitment to a compliance store, queryable before client confirmation. Neither definition was wrong in its own context. The gap was that no process existed to reconcile these meanings across boundaries. **No cross-referencing between ADRs and domain rules.** ADR-014 was reviewed in an architecture silo. Bird's domain rules lived in a domain model document. There was no mechanism — automated or manual — to check whether a new architectural decision was compatible with existing domain constraints. The two artifacts existed in parallel but never intersected. **Kobe's review checklist scoped to architecture, not compliance.** Kobe's review was thorough within its defined scope. The checklist explicitly covered scalability, failure modes, operational complexity, and security. It did not cover regulatory compliance or domain rule compatibility. This scoping made sense for most ADRs — but for ADRs touching regulated data flows, it left a blind spot. **Bird not tagged as a reviewer on ADR-014.** There was no policy or automation to identify when an ADR touches a regulated domain and route it to Bird for domain compliance review. Bird's participation would very likely have caught the mismatch, but the review process had no trigger to include Bird. **No audit trail completeness monitoring.** Pippen's operational monitoring did not include a check for projection completeness relative to regulatory query windows. The 200ms lag was monitored as a performance metric, not as a compliance risk. The absence of this monitoring meant the gap was invisible for 4 months — discovered externally rather than internally. **The deprioritization of compliance monitoring was reasonable at the time.** Before ADR-014, order state was written synchronously — there was no projection lag to monitor. When the architecture changed, no one updated the monitoring requirements because the projection lag was understood as a performance characteristic, not a compliance-relevant property. The mental model of "what needs monitoring" did not update when the architecture changed. ## 4. Learnings **We now know that "durable" means different things in architecture and regulatory contexts**, which means any ADR touching data that flows to regulators must explicitly define its durability semantics in regulatory terms, not just infrastructure terms. **We now know that domain rules and architecture decisions can drift silently when reviewed in separate streams**, which means we need a cross-referencing mechanism that forces intersection at review time, not after an incident. **We now know that Kobe's review checklist has an implicit assumption that reviewed systems do not touch regulated flows**, which means the checklist needs a triage step that identifies regulatory exposure and routes to Bird when present. **We now know that changing a write path from synchronous to asynchronous changes the compliance properties of the system, not just the performance properties**, which means any sync-to-async migration for data subject to regulatory requirements must be treated as a compliance-affecting change. **We now know that the absence of audit trail completeness monitoring made us reliant on external auditors to find our gaps**, which means detection capability for compliance-relevant projection lag is not optional — it is as critical as the write path itself. ## 5.
