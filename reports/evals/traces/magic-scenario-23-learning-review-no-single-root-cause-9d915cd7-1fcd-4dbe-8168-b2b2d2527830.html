# Learning Review: Payment Service Double-Charge During Infrastructure Failover **Date of incident**: 2026-03-05 **Review prepared by**: Magic (Context Synthesizer) **Input sources**: Shaq (implementation), MJ (architecture), Kobe (operations), Bird (domain), Pippen (observability) --- ## 1. Situation Snapshot During a scheduled infrastructure failover on 2026-03-05, the payment processing service double-charged 312 customers, totaling £28,400 in duplicate charges. The root mechanism was an idempotency key collision caused by a counter-service fallback that reset attempt numbers during the failover window. The incident affected customers for approximately 40 minutes before detection (via customer complaint, not alerting). Refunds were issued within 6 hours of detection. All five team perspectives — implementation, architecture, operations, domain, and observability — contributed independent conditions that were each necessary for this outcome. --- ## 2. Timeline | Time | Event | What was known | Decision made | |------|-------|---------------|---------------| | T+0:00 | Scheduled infrastructure failover begins | Ops team follows established runbook | Restart services in documented dependency order | | T+0:00–T+~3:00 | Payment service restarts before counter service | Counter service classified as "non-critical" in runbook | Payment service comes online, cannot reach counter service | | T+0:00–T+4:00 | 312 payment requests hit counter-reset fallback | Payment service logs show fallback activation (not monitored) | Fallback silently resets attempt_number to 1, generates new idempotency keys | | T+4:00 | Failover completes, counter service back online | All services healthy | No anomaly detected | | T+~44:00 | Customer contacts support about double charge | First signal that something went wrong | Support investigates, discovers pattern | | T+~50:00 | Engineering confirms 312 affected customers | Scope of double-charges identified | Refund process initiated | | T+6:00:00 | All refunds issued | Full customer list confirmed | Incident closed for customer impact | --- ## 3. Contributing Factors Each of the following conditions was independently necessary for this incident to occur. Remove any single one and the double-charges do not happen. **Factor 1: The counter-service fallback resets attempt_number to 1.** Shaq implemented this fallback 8 months prior as a defensive measure to keep payments flowing if the counter service was temporarily unreachable. This was a reasonable resilience decision in isolation — letting payments fail entirely during a brief network blip would have had its own customer impact. The fallback's failure mode (generating a previously-unused idempotency key) was not obvious at implementation time. **Factor 2: The idempotency key design does not encode enough uniqueness to survive a counter reset.** The key formula (order_id + attempt_number) produces identical keys when attempt_number resets for the same order. MJ's architecture did not account for the possibility that the counter could reset, because the counter service was treated as a reliable singleton. A key design that included a unique component (e.g., a timestamp, UUID, or node ID) would have produced distinct keys even after a reset. **Factor 3: Stripe's 24-hour idempotency window did not catch the collision.** Stripe deduplicates charges with the same idempotency key within a 24-hour window. This safety net would have prevented the double-charge if the reset had produced the *same* key as the original attempt. But because the counter reset to 1 for a *new* payment attempt on the same order, the generated key was one that had never been submitted to Stripe before. The external safety net was architecturally irrelevant to this failure mode. **Factor 4: The failover runbook classified the counter service as non-critical.** Kobe's review confirmed the runbook listed the counter service for last-in-line restart. This classification was made without knowledge of the payment service's idempotency dependency on the counter service. The runbook had been updated 3 months prior for a different dependency, demonstrating that the process for documenting dependencies existed but was applied incompletely. The dependency was not obvious — a "counter" service does not naturally signal "payment correctness depends on me." **Factor 5: The payment service's acceptance criteria did not cover counter-service-unavailable scenarios.** Bird confirmed that the domain rule ("a customer must never be charged more than once for a single order") was well understood, but its technical implementation had an untested failure mode. The acceptance criteria validated the happy path of idempotency but not the degraded-dependency path. This is a common gap: domain rules are tested against the system-as-designed, not against the system-as-degraded. **Factor 6: No monitoring existed for idempotency key uniqueness or duplicate charge detection.** Pippen confirmed that observability covered charge success/failure rates but not duplicate-charge patterns. The 312 double-charges registered as 312 successful charges — a metric that looks healthy. There was no alert on "two successful charges for the same order within N minutes" or on "counter-service fallback activated." The incident was detected by a customer complaint 40 minutes later, not by any automated system. **Factor 7: The fallback was never tested against a full infrastructure failover scenario.** Shaq noted the fallback was designed for brief counter-service unavailability, not a coordinated failover where the counter service is deliberately offline for minutes while the payment service is actively processing. The gap is understandable — testing against full failover scenarios is expensive and the fallback was a pragmatic addition, not a formally specified feature. --- ## 4. Learnings - **We now know that resilience fallbacks can violate domain invariants**, which means every fallback path in a payment-critical service needs explicit analysis against the domain rules it might weaken. A fallback that "keeps things working" can be worse than a fallback that fails loudly if "working" means silently violating a financial invariant. - **We now know that idempotency key design must be robust to component restarts and counter resets**, which means keys should include enough entropy or uniqueness (timestamp, UUID, node identifier) to remain collision-free even when upstream state is lost. Relying on a monotonic counter from another service creates a hidden coupling. - **We now know that the failover runbook's dependency model was incomplete**, which means service dependency documentation needs to capture not just "Service A calls Service B" but also "Service A's correctness property X depends on Service B's state Y." Operational dependencies are a superset of runtime dependencies. - **We now know that monitoring charge success rates is insufficient to detect duplicate charges**, which means observability for payment systems must include domain-level invariant checks, not just operational health metrics. A double-charge looks like two successes. - **We now know that the domain team (Bird) was unaware of the fallback's existence**, which means changes to payment-critical code paths that introduce degraded modes need domain review, not just architectural and code review. The people who own "a customer must never be charged twice" need to know about every code path that could violate that rule. --- ## 5.
