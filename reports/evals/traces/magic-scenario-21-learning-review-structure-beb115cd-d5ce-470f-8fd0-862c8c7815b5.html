# Learning Review: Stale Seasonal Pricing Due to Cache TTL Misconfiguration **Incident date**: 2026-03-12 | **Review date**: 2026-03-24 --- ## 1. Situation Snapshot On 2026-03-12, a cache invalidation change (pricing-service v2.4.1, PR #1847) was deployed by Shaq and reviewed by Kobe. The change altered how TTL values are computed for category-specific Redis cache entries, inadvertently bypassing the seasonal category's 15-minute TTL cap during an active promotional window. For approximately 20 minutes (14:22–14:42 UTC), 847 customer transactions on seasonal products were charged stale promotional prices from the previous day's batch. Pippen's price divergence alert detected the anomaly at 14:38 UTC, and rollback was completed four minutes later. --- ## 2. Contributing Factors Each of these factors independently contributed. No single one is "the root cause" — the incident required all of them to coincide. 1. **The TTL computation was moved from a config-driven lookup to a call-site parameter, silently decoupling it from CategoryConfig overrides.** The new pattern worked correctly for any category without a TTL override. The seasonal category's 15-minute cap was encoded in CategoryConfig but was no longer consulted by the new code path. 2. **Staging fixtures did not include seasonal products.** Shaq tested the change against the "standard" category in staging. This was not negligent — staging fixtures had been sufficient for prior pricing-service changes, and no one had flagged the seasonal gap before. The fixture set simply reflected the most common product categories, not the full domain. 3. **The ticket was scoped as "performance optimization," which did not trigger domain review.** Bird, who holds authority over pricing domain rules, was not consulted during spec or review. This scoping made sense at the time — the change was genuinely motivated by performance, and the team's process only routes pricing-rule changes to Bird. The problem is that a change can be *motivated* by performance while still *affecting* pricing rules. 4. **Code review focused on logical correctness of the changed path, not on domain invariants.** Kobe verified the standard-category path and unit test coverage. The seasonal TTL override lives in CategoryConfig, which is a domain-level constraint — not something that naturally surfaces during a diff-focused code review. Kobe's review was thorough for what the PR presented; the issue was in what it *didn't* present. 5. **No automated test validates that cache TTLs conform to CategoryConfig domain rules.** The seasonal TTL cap is a domain invariant, but it was enforced only by the old code path's structure (reading from CategoryConfig), not by an explicit test. When the structure changed, the invariant was silently violated. 6. **Price divergence alerting has a 5% threshold, introducing a detection delay.** The alert fired 16 minutes after deployment. This is a reasonable threshold for catching pricing anomalies without excessive noise, but it means that time-sensitive pricing errors accumulate customer impact before detection. --- ## 4. Learnings - **We now know that a change's *motivation* (performance) can differ from its *domain impact* (pricing rules).** This means ticket scoping based on intent alone is insufficient to determine who needs to review a change. Changes that touch domain-configured values need domain review regardless of why they are being changed. - **We now know that CategoryConfig contains domain invariants that are load-bearing but invisible to tests.** This means domain rules encoded only in configuration — without corresponding automated assertions — are fragile to structural refactors. If a rule matters, it must be tested independently of the code path that currently enforces it. - **We now know that staging fixture gaps can hide category-specific regressions.** This means our staging environment gives false confidence for any code path that diverges by product category, customer segment, or similar domain partitions. Fixture coverage must reflect domain partitions, not just common-case data. - **We now know that diff-focused code review is insufficient for catching domain invariant violations.** This means that when a change restructures how a value is sourced (even if the new logic looks correct in isolation), the reviewer needs a way to see which domain constraints depended on the old sourcing pattern. This is a tooling and process problem, not a reviewer diligence problem. --- ## 5.
