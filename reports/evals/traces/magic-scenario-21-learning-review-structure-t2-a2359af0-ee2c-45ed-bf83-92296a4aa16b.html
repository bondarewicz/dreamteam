# Learning Review: Seasonal Pricing Cache Staleness Incident (2026-03-12) ## Situation Snapshot On 2026-03-12, a cache invalidation refactor (PR #1847) was deployed to the product pricing service by Shaq and approved by Kobe. The change altered how TTL values are computed for category-specific cache entries, moving from a config-driven lookup to a call-site-passed value. This inadvertently bypassed the seasonal category's 15-minute TTL ceiling — a domain rule protecting time-sensitive promotional pricing. For 20 minutes (14:22–14:42 UTC), 847 customer transactions on seasonal items were charged stale prices from the previous day's promotional batch. Pippen's price divergence alert detected the anomaly at 14:38 UTC, and rollback completed four minutes later. ## Timeline | Time (UTC) | Event | What was known | Decision / Action | |---|---|---|---| | Pre-deploy | PR #1847 scoped as "performance optimization" for cache invalidation. | Team understood this as a non-functional change to how TTL is passed. The rollback playbook worked as designed. | ## Contributing Factors No single cause produced this incident. The following independent factors had to co-exist: 1. **Ticket framing excluded domain review.** The cache invalidation work was scoped as a "performance optimization," which placed it outside Bird's domain review path. The seasonal TTL ceiling is a domain rule, but the ticket's categorization created no trigger for domain consultation. This framing was reasonable at the time — cache TTL mechanics genuinely feel like infrastructure, not business logic. 2. **Domain rules encoded in config, not enforced in code.** The 15-minute seasonal TTL ceiling lives in the `CategoryConfig` enum but is not enforced by an invariant or guard clause at the cache write boundary. The rule is a convention, not a constraint. Any code path that bypasses the config lookup can silently violate it. 3. **The refactor changed the TTL authority from config-read to call-site-pass.** This is the mechanical cause — TTL was previously looked up from `CategoryConfig` at cache write time, guaranteeing category-specific rules were honored. The new pattern passes TTL from the call site, which worked for categories without overrides but silently skipped the seasonal ceiling. The pattern was correct in structure but wrong in assumption: it assumed all categories use the same TTL source. 4. **Staging fixture set did not include seasonal products.** Shaq tested in staging, but the fixture data only covered the "standard" category. The seasonal category — the one with the special TTL rule — was never exercised. This gap was not documented or visible in any test coverage report. 5. **Code review focused on structural correctness, not domain rule compliance.** Kobe verified that the refactored code was internally consistent and that unit tests passed. He was aware of the seasonal TTL override's existence but did not trace the new code path to confirm it still read that override. This is understandable — code review typically validates what the PR claims to do, and the PR did not claim to change pricing behavior. 6. **No automated check links cache TTL values to domain rules.** There was no integration test, contract test, or policy check that compared actual cache TTLs against the `CategoryConfig` domain rules. The divergence was only detectable by the price-feed comparison alert, which has inherent latency (the 5% threshold takes time to breach). ## Learnings - **We now know that cache TTL values are domain rules, not just infrastructure knobs**, which means any change to TTL computation must trigger domain review regardless of how the ticket is framed. The boundary between "performance" and "business logic" is not always obvious — TTL ceilings that protect promotional pricing are pricing rules wearing infrastructure clothing. - **We now know that our staging fixture set has blind spots for category-specific behavior**, which means staging confidence is only as good as fixture coverage. A green staging run against "standard" tells us nothing about "seasonal." - **We now know that domain rules in config enums are advisory, not enforced**, which means any refactor that changes how config is read can silently bypass domain constraints without any test or runtime failure. - **We now know that Pippen's price divergence alert works and catches real issues**, which means the detection layer is sound — but 16 minutes of exposure is too long for promotional pricing. The alert proved its value; the question is whether we can tighten the window. - **We now know that ticket scoping directly controls who reviews the work**, which means a miscategorized ticket can route critical changes past the people best equipped to catch problems. Bird's absence from this review was a natural consequence of the ticket's framing, not negligence by anyone. ## Action Items 1. **`[PREVENT]` Add a domain-enforced TTL guard at the cache write boundary.** The cache layer must read and enforce category-specific TTL ceilings from `CategoryConfig` regardless of what TTL the call site passes. Due: 2026-04-07.** 4. **`[PREVENT]` Add seasonal products to the staging fixture set.** Ensure staging includes representative data for every pricing category, especially those with domain-specific overrides.
