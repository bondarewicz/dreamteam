A courier marketplace wants to build a real-time courier bidding system: - When an order is placed, the system sends the job to all eligible couriers simultaneously - Couriers (via mobile app) have 15 seconds to accept or the job is auto-assigned to the highest-rated available courier - During the 15-second window, courier bids (accepting the job) must be processed in real time - The system must handle 500 simultaneous active bid windows (500 orders in the bidding phase at once) - The first courier to accept wins — acceptance is atomic (two couriers cannot both accept the same job) - Average courier pool per zone: 30 couriers Design the bidding system architecture. Let me analyze this courier bidding system. This is a well-constrained problem with clear NFRs. Let me classify and design. Key constraints: - Real-time bidding with 15-second windows - 500 concurrent bid windows - Atomic acceptance (first-write-wins) - 30 couriers per zone average - Auto-assign fallback to highest-rated courier - Mobile app clients This is a classic distributed systems problem involving: - Real-time push notifications - Optimistic concurrency / atomic operations - Timer-based state transitions - WebSocket or SSE for real-time communication Now I'm thinking through the scale: 500 concurrent windows with 30 couriers each gives me around 15,000 concurrent WebSocket connections, which is manageable. For the bidding throughput, worst case is 15,000 bids per second in a burst, though realistically it'll be much lower. I'll use Redis with atomic operations like SETNX or Lua scripts to handle the first-write-wins semantics, WebSockets to push bids to couriers in real-time, and a scheduled timer to manage the 15-second window expiry with event-driven processing for anything downstream.
