Key findings: 1. **JWT_SECRET with non-null assertion** - `process.env.JWT_SECRET!` will throw if undefined, but worse, if it's an empty string, jwt.verify may behave unexpectedly. No fallback or startup validation. 2. **No authorization check on /deliveries/:id** - Authentication exists but no authorization. Any authenticated user can view ANY delivery by ID. This is an IDOR vulnerability. 3. **Location endpoint has no rate limiting or input validation** - Public endpoint with no auth could be abused. The location endpoint is completely unauthenticated despite exposing real-time delivery locations—the comment claims location data is public for tracking purposes, but this actually allows anyone to enumerate sequential IDs and access live location data for all deliveries. The three most critical issues are the IDOR vulnerability on the delivery endpoint, the unauthenticated location endpoint leaking real-time tracking data, and the unsafe JWT_SECRET handling that could crash the server or fail silently.
