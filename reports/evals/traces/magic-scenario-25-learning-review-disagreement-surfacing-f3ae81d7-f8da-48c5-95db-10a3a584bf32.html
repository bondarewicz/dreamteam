# Learning Review: user_preferences Migration Partial Failure — 2026-02-14 ## Situation Snapshot During a scheduled maintenance window on 2026-02-14, a database migration adding a nullable JSON column to the `user_preferences` table (800k rows) failed at 60% completion due to a deadlock between the backfill UPDATE and a live API transaction. The team disagreed on the recovery path — rollback vs. forward-fix — and ultimately chose rollback, which required replaying 4 hours of accumulated preference events. The preference API served stale (pre-migration) data for 47 minutes during recovery. No data was permanently lost. ## Timeline 1. **Maintenance window opens.** Migration begins: an ALTER TABLE adds a nullable JSON column, followed by a single-statement backfill UPDATE across all 800k rows. The team expects this to complete within the window. No batching is used; the runbook does not specify a threshold. 2. **~11 minutes in, backfill fails.** A deadlock occurs between the backfill UPDATE and a live API transaction performing a SELECT FOR UPDATE on the same table. The backfill has completed ~60% of rows. At this moment, the team knows the table is in a partially-migrated state — some rows populated, others null. 3. **Disagreement emerges.** Kobe advocates immediate rollback to a known-good state, reasoning that the API tolerates a missing column and that the deadlock risk is not yet understood. MJ advocates forward-fix, reasoning that rollback introduces an unrehearsed event replay operation and that the already-migrated rows are consistent. Both are reasoning from first principles — no pre-agreed framework exists for this decision. 4. **Team chooses rollback.** The partially-migrated rows are reverted. The column addition is reversed. The database returns to its pre-migration schema. 5. **Event replay required.** During the migration attempt and rollback, 4 hours of preference update events accumulated in the event log without being applied. Designing and executing the replay takes ~4 hours of engineering effort and 47 minutes of execution time. 6. **Service restored.** Replay completes successfully. No data is lost. However, for the 47-minute replay window, the preference API served pre-migration data. ## Contributing Factors Each of the following factors independently contributed to the outcome. No single factor is "the root cause." **1. The backfill was a single unbatched UPDATE across 800k rows.** This maximized lock duration and the window for deadlock with concurrent transactions. Shaq followed the runbook, which did not specify a batch size threshold. The decision to skip batching was reasonable given the information available — 800k rows is not obviously large enough to require batching in all database engines, and the runbook was silent on the topic. **2. The maintenance window did not fully isolate the table from live traffic.** The API continued to perform SELECT FOR UPDATE operations against `user_preferences` during the migration. Whether this was intentional (to minimize downtime) or an oversight is unclear, but it created the conditions for the deadlock. **3. The semantic ambiguity of a NULL value in the new column was not resolved before migration approval.** Bird's domain rule — preference data must be consistent per user, partial states are not permitted — was in tension with the migration's intermediate state where NULL could mean either "no preference set" or "not yet migrated." This ambiguity was load-bearing during the incident: it affected how dangerous the 40%-migrated state actually was, which in turn affected whether rollback or forward-fix was the right call. But the team had not surfaced this ambiguity before the migration. **4. No pre-agreed decision framework existed for partial migration failure.** Both Kobe and MJ noted this explicitly. The team had no shared criteria for when to roll forward vs. roll back, no decision tree, no pre-assigned authority. This meant the disagreement had to be resolved in real time under pressure, consuming time and cognitive load. **5. The event replay operation had never been rehearsed.** MJ's concern about replay correctness risk was well-founded — the replay was a novel operation designed under pressure. The fact that it succeeded does not retroactively make it low-risk; it means the team got it right despite the pressure. **6. The runbook did not account for partial failure scenarios.** The runbook covered the happy path. It did not specify what to do if the backfill failed mid-run, did not define a batch size threshold, and did not address concurrent transaction risks. ## The Kobe/MJ Disagreement as a Signal The disagreement between Kobe and MJ was not a failure of judgment by either party. Both positions were internally coherent: - **Kobe's position** prioritized returning to a known-good state and avoiding continued exposure to an incompletely-understood deadlock. This is a defensible stance rooted in the principle that uncertainty about failure modes should bias toward reverting. - **MJ's position** prioritized avoiding the introduction of a new, unrehearsed operation (event replay) and preserving work already completed. This is a defensible stance rooted in the principle that rollback is not free — it has its own risks, and those risks were less well-understood than the forward path. The fact that these two experienced team members reached opposite conclusions from the same information is the most important signal from this incident. It reveals that **the team did not have a shared mental model for evaluating partial migration states.** Specifically: - There was no shared agreement on whether a partially-migrated table is "closer to safe" or "closer to dangerous." - There was no shared agreement on whether rollback or forward-fix carries more risk when the failure mode is a deadlock. - There was no shared understanding of the cost of event replay, because it had never been estimated or rehearsed. This is not a disagreement to resolve retroactively by picking a winner. It is a gap in shared understanding to close proactively, so the next time a partial failure occurs, the team has pre-agreed criteria rather than a real-time debate. ## Learnings 1. **We now know that a single unbatched UPDATE against a table with concurrent live traffic creates a wide deadlock window,** which means migration backfills should default to batched execution unless there is a specific reason not to batch. 2. **We now know that NULL semantics in a new column must be explicitly defined before migration approval,** which means domain review (Bird) should include a sign-off on "what does NULL mean during and after migration?" as a gate. 3. **We now know that the team has no shared decision framework for partial migration failure,** which means rollback-vs-forward-fix decisions are currently made ad hoc under pressure. This is the highest-leverage gap to close. 4. **We now know that event replay, while successful, was an unrehearsed operation that took 4 hours to design,** which means our recovery playbooks underestimate the cost of rollback when event logs are involved. 5. **We now know that "maintenance window" did not mean "exclusive table access,"** which means the team's definition of maintenance window isolation needs to be explicit — does it include connection draining, read-only mode, or full lock? ## Action Items - `[PREVENT]` **Add a "partial failure runbook" section to the migration template** that specifies: (a) batch size guidance, (b) rollback-vs-forward criteria, (c) who has decision authority. **Owner: Pippen.
