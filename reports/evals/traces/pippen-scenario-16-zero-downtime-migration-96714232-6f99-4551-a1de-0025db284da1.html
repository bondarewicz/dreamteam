You are Pippen reviewing a zero-downtime migration plan for a "user preferences" feature. The migration moves user preferences from a JSON blob column in the users table to a dedicated user_preferences table. Migration plan: Step 1: Add user_preferences table (no data, just schema) Step 2: Deploy new application code that writes preferences to BOTH users.preferences_json AND user_preferences table (dual-write) Step 3: Run backfill job: copy existing preferences from users.preferences_json to user_preferences for all existing users Step 4: Deploy new application code that reads preferences from user_preferences only (stops reading from users.preferences_json) Step 5: Stop dual-write — write only to user_preferences Step 6 (future): Drop the users.preferences_json column Service details: - Multiple instances of the application run simultaneously (horizontal scaling, 3 instances) - The deployment in Step 4 is a rolling deployment — instances update one at a time - No feature flag to coordinate the read source switch across all instances simultaneously - Backfill job has no progress tracking or resumability - No validation step to confirm backfill completeness before Step 4 Assess the migration plan's operational readiness.

Let me analyze this migration plan for operational readiness issues. Key issues I see: 1. Step 3→4 gap: No validation that backfill is complete before switching reads 2. Step 4 rolling deployment: During rollout, some instances read from old source, some from new — if backfill incomplete, new-reader instances serve stale/missing data 3. Backfill job: No progress tracking, no resumability — if it fails midway, you don't know where it stopped 4. No feature flag: Can't coordinate read switch atomically across instances 5. Race condition: During Step 3 backfill, new writes go to both tables (dual-write), but what about rows being backfilled that get updated concurrently? The backfill could overwrite a newer dual-written value with an older one from the JSON blob. 6. There's a critical data loss scenario here — if a user updates their preferences after backfill starts reading their old data, the backfill will write stale values back and overwrite the concurrent dual-write with older information. 7. No rollback plan documented, and dropping the column in Step 6 is irreversible.
