# Learning Review: Invoicing Service Supplier ID Mismatch Incident **Date of review**: 2026-03-24 **Incident window**: 2026-02-28 to 2026-03-03 (3 days undetected) --- ## 1. Situation Snapshot The Invoicing Service, which ingests supplier invoice data via an external webhook, began persisting malformed `supplier_id` values on 2026-02-28. Some supplier IDs arrived as float-formatted strings (e.g., `"12345.0"`) instead of the expected integer-formatted strings (`"12345"`). The records were written successfully — no errors were raised — but a downstream reconciliation job that performs exact-string matching on `supplier_id` silently failed to match these records. The discrepancy went undetected for 3 days and was discovered by the finance team during month-end close, resulting in a £47,000 gap in the monthly reconciliation report. No data was lost, but trust in the reconciliation pipeline was materially damaged. --- ## 2. No contract test or schema validation catches it. | | 2026-02-28 | Float-formatted `supplier_id` values begin arriving and are persisted without error. | Reconciliation job starts logging WARNINGs for unmatched IDs. Investigation traces the gap to `supplier_id` format mismatch. | --- ## 3. Contributing Factors Each of the following factors independently contributed to this outcome. Removing any single one would likely have prevented or significantly reduced the impact. 1. **No input normalisation on `supplier_id` at the ingestion boundary.** The system accepted and persisted whatever the webhook sent, with no canonicalisation step. Bird had identified this gap and recommended a fix (TD-291), but it had not yet been implemented. 2. **The deprioritisation of TD-291 outlived its original risk assessment.** The decision to defer was sound at the time — the provider had confirmed integer-formatted IDs, and the team was under schedule pressure. However, there was no mechanism to re-evaluate the risk when conditions changed (e.g., new sub-provider onboarding). The accepted risk had no expiry date or review trigger. 3. **The webhook provider made an unannounced upstream change.** A new sub-provider was onboarded whose system emits float-formatted IDs. This was not communicated. The team had no contract test or schema validation at the integration boundary that would have detected the format change independently of the provider's communications. 4. **The reconciliation job treated match failure as a warning, not an alert.** The job logged `WARNING` for unmatched `supplier_id` values but did not escalate. The warning volume was low enough to stay below the manual review threshold, meaning the signal existed but was effectively invisible. 5. **Exact-string matching in reconciliation was brittle to semantically-equivalent but syntactically-different IDs.** `"12345"` and `"12345.0"` refer to the same supplier but were treated as different values. The reconciliation logic had no tolerance for format variation, making it a silent-failure point for any upstream format drift. --- ## 4. Learnings - **We now know that accepted risk items need a review trigger, not just a backlog slot.** TD-291 was logged as tech debt but had no mechanism to force re-evaluation when the external integration landscape changed. This means that any "risk accepted" decision should include a concrete condition or date that triggers re-assessment — especially for data quality risks at integration boundaries. - **We now know that warnings without alerting thresholds are effectively silent.** The reconciliation job was logging the exact signal that would have caught this incident on day one, but it was configured as a warning with no alert. This means that any job producing financial output needs its warning signals to escalate automatically — the finance team should not be the first line of detection for reconciliation failures. - **We now know that external webhook contracts are not stable unless we enforce them ourselves.** The provider confirmed integer formatting, but a sub-provider change invalidated that guarantee without notice. This means that the only reliable contract enforcement is at our own ingestion boundary — we cannot delegate format guarantees to external parties. - **We now know that exact-string matching on externally-sourced identifiers is inherently fragile.** Semantically identical values (`"12345"` vs `"12345.0"`) can diverge syntactically at any time. This means that any matching or joining logic on externally-sourced IDs should either normalise inputs before storage or use a tolerant matching strategy. --- ## 5. Action Items 1. **`[PREVENT]` Implement `supplier_id` normalisation at the webhook ingestion boundary.** Strip trailing `.0`, remove zero-padding, enforce canonical integer-string format before persistence. Retroactively repair affected records. **Owner: Shaq. Due: 2026-04-04.** 2. **`[DETECT]` Add an alert on the reconciliation job's unmatched-ID warning count.** If unmatched `supplier_id` count exceeds 0 in any run, fire an alert to the engineering on-call channel. Zero tolerance — even one mismatch in financial reconciliation is worth investigating. **Owner: Pippen. Due: 2026-04-04.** 3. **`[DETECT]` Add a contract test at the webhook ingestion boundary.** Validate that incoming `supplier_id` values match the expected canonical format (integer string, no decimals, no padding). Log and alert on any value that requires normalisation — this is an early-warning signal for upstream changes. **Owner: Shaq. Due: 2026-04-11.** 4. **`[PROCESS]` Introduce a "risk accepted" review cadence for deferred tech debt items at integration boundaries.** Any TD item tagged with `external-integration` and `risk-accepted` must be reviewed at the start of each sprint. If the external system's conditions have changed (new providers, new contract terms), the item is automatically re-prioritised. **Owner: Coach K. Due: 2026-04-11.** 5. **`[MITIGATE]` Add a reconciliation pre-check that flags semantically-equivalent but syntactically-different ID matches.** Before producing the final reconciliation report, run a fuzzy-match pass that identifies likely matches (e.g., `"12345"` vs `"12345.0"`) and surfaces them for review rather than silently dropping them. **Owner: Shaq. Due: 2026-04-18.** 6. **`[DETECT]` Request a change-notification clause from the webhook provider.** Require that the provider notify us of any upstream changes to sub-provider systems, ID formats, or data schemas with a minimum lead time. Preserving the Learning - **Domain rule (Bird to codify):** "`supplier_id` is an opaque string identifier that MUST be normalised to canonical integer-string format at the ingestion boundary before persistence. Any external-sourced identifier used in downstream matching or joining logic must pass through a normalisation step." This rule should be added to the Invoicing Service's domain documentation and enforced by the contract test in action item 3. - **ADR (Magic to draft):** "ADR: Input normalisation required at all external integration boundaries." Documents the decision that all externally-sourced identifiers must be normalised at ingestion, with the rationale drawn from this incident. Zero-match-failure tolerance for reconciliation jobs." - **Tech debt process update (Coach K to own):** Update the team's tech debt triage template to include a "Risk review trigger" field. Any deferred item at an external integration boundary must specify the condition under which it should be automatically re-evaluated.
