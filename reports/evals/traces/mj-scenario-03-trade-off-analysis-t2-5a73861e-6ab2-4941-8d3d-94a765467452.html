There are several architectural concerns: - Write throughput: If you have many couriers updating every few seconds, PostgreSQL can handle this but it's not ideal - Read latency: Querying PostgreSQL for every customer view adds latency - Staleness enforcement: Can be done at query level (WHERE timestamp > now() - 30s) but this is application-level, not infrastructure-level - The 30-second rule is strict - "must not be displayed" - this is a hard domain invariant The real question is whether PostgreSQL can actually satisfy the 30-second freshness requirement. LISTEN/NOTIFY could bridge that gap, but it has limitations. The core tension is that Bird's domain rule demands true real-time semantics with strict staleness enforcement, while a PostgreSQL-first approach relies on polling and query-time validation—which technically works but doesn't naturally align with push-based real-time delivery.
