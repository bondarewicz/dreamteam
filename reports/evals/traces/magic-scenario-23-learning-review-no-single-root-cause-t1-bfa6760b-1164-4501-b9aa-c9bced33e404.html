# Learning Review: Payment Service Double-Charge During Infrastructure Failover **Date of incident**: 2026-03-05 **Review produced**: 2026-03-26 **Scope**: 312 customers double-charged, £28,400 total exposure, refunds completed within 6 hours --- ## 1. Situation Snapshot During a scheduled infrastructure failover on March 5th, the payment processing service double-charged 312 customers over a 4-minute window. The root mechanism was an idempotency key collision: when the counter service was unreachable during failover, the payment service's fallback logic reset the attempt counter to 1, generating idempotency keys identical in structure but semantically new to Stripe. The duplicate charges were detected by a customer complaint 40 minutes post-failover, not by internal monitoring. Refunds were issued within 6 hours. Five team members across implementation, architecture, operations, domain, and observability contributed post-incident analysis. --- ## 2. Timeline | Time | Event | What was known | Decision made | |------|-------|----------------|---------------| | T+0:00 | Scheduled infrastructure failover begins | Ops team follows runbook; failover is routine | Proceed per documented runbook | | T+0:00–T+1:00 | Services restarted in runbook-defined dependency order | Counter service classified as non-critical | Counter service scheduled for last restart | | T+~1:00 | Payment service comes online, cannot reach counter service | Payment service logs show counter-service-unavailable; fallback activates | Fallback designed to keep payments flowing — working as implemented | | T+1:00–T+4:00 | 312 payment requests processed with reset attempt_number=1 | No one is watching idempotency key uniqueness; charge success metrics look normal | No intervention — nothing appears wrong | | T+4:00 | Counter service comes online; failover completes | All services healthy; failover marked successful | — | | T+~44:00 | Customer contacts support reporting double charge | First signal that something went wrong | Support escalates; engineering begins investigation | | T+~60:00 | Engineering confirms 312 duplicate charges via database query | Scope and mechanism understood | Refund process initiated | | T+6:00:00 | All 312 refunds issued | — | Incident closed operationally | --- ## 3. Contributing Factors Each of the following conditions was independently necessary for this outcome. Remove any one, and the double-charges do not occur. **Factor 1: The counter-reset fallback produced structurally valid but semantically colliding idempotency keys.** The payment service's fallback, when unable to reach the counter service, reset `attempt_number` to 1. This generated an idempotency key (`order_id + 1`) that was syntactically valid and passed all local validation — but was identical to the key used by the original, already-completed charge. The fallback was a reasonable defensive choice at the time: keeping payments flowing during brief counter-service blips. The scenario where it would produce a key collision (counter unavailable *after* a successful first charge) was not anticipated. **Factor 2: The idempotency key design coupled correctness to counter-service availability without documenting this coupling.** The idempotency scheme (`order_id + attempt_number`) implicitly assumed the counter service would always provide a monotonically increasing attempt number. This assumption was not documented as an invariant. When the assumption was violated, the key generation silently produced duplicates rather than failing loudly. **Factor 3: Stripe's 24-hour deduplication window did not catch the collision.** Stripe deduplicates charges by idempotency key within a 24-hour window — if the *same* key is resubmitted, Stripe returns the original result. However, the counter reset meant the second charge used a key that Stripe had never seen (the original charge used the real attempt number from the counter service; the fallback charge used `1`). The external safety net existed but was architecturally bypassed by the key collision mechanism. **Factor 4: The failover runbook classified the counter service as non-critical, restarting it after the payment service.** The runbook's dependency ordering did not reflect the payment service's reliance on the counter service for idempotency correctness. The counter service *is* non-critical for many other operations — the classification was not wrong in general, only incomplete. A dependency note for a different service had been added to the runbook 3 months prior, showing the process for updating it existed but had not been applied here. **Factor 5: No monitoring existed for idempotency key uniqueness or duplicate charge detection.** Observability covered charge success/failure rates but not whether multiple successful charges occurred for the same order. The double-charge pattern was invisible to all existing dashboards and alerts. Detection came from a customer complaint, not from the system. **Factor 6: The acceptance criteria for the payment service did not include counter-service-unavailable scenarios.** The domain rule ("a customer must never be charged more than once for a single order") was not tested under degraded-dependency conditions. The acceptance criteria validated the happy path and explicit failure paths, but not the partial-availability path where the fallback activates. **Factor 7: The fallback was never tested against a full infrastructure failover scenario.** The fallback was implemented 8 months prior. It was likely tested against brief counter-service unavailability in isolation, but not against the specific scenario of a failover where the payment service processes real traffic while the counter service is still down. The gap between "unit-tested fallback" and "failover-tested fallback" went unnoticed for 8 months. --- ## 4. Learnings - **We now know that fallback logic in payment paths can silently violate domain invariants**, which means every fallback in a money-touching service needs explicit analysis of what invariants it relaxes and whether that relaxation is acceptable. - **We now know that idempotency key correctness depends on the availability of the counter service**, which means the counter service is a critical dependency for payment correctness even though it is not critical for payment *processing*. Criticality is not a single dimension. - **We now know that Stripe's deduplication is a safety net only when the same key is resubmitted**, which means any mechanism that generates a *new* key for a *repeated* operation bypasses external deduplication entirely. External safety nets have preconditions we must understand. - **We now know that our observability has a gap between "charge succeeded" and "charge was correct"**, which means success-rate monitoring is necessary but not sufficient for payment integrity. - **We now know that runbook dependency classifications decay as the system evolves**, which means dependency documentation needs a mechanism to stay current when new couplings are introduced (like the fallback PR 8 months ago). - **We now know that domain invariants need to be tested under degraded-dependency conditions**, which means acceptance criteria that only cover the happy path and full-failure path miss the most dangerous scenario: partial availability where the system *appears* to work correctly. --- ## 5.
