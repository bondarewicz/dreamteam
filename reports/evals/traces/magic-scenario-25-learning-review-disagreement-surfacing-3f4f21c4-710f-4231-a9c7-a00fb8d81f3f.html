# Learning Review: user_preferences Migration Deadlock and Recovery — 2026-02-14 ## Situation Snapshot During a scheduled maintenance window on 2026-02-14, a database migration adding a nullable JSON column to the `user_preferences` table (800k rows) failed at 60% completion due to a deadlock between the backfill UPDATE and a live API `SELECT FOR UPDATE`. The team disagreed on recovery path — Kobe advocated rollback, MJ advocated forward-fix — and ultimately rolled back, requiring a 4-hour event log replay. The preference API served degraded (pre-migration) data for 47 minutes while the replay executed. No data was lost. ## Timeline 1. **Migration begins** — Shaq applies the migration per the runbook: adds a nullable JSON column, then kicks off a single unbatched `UPDATE` to backfill all 800k rows from the event log. 2. **~11 minutes in — deadlock** — The backfill UPDATE collides with a live API transaction performing a `SELECT FOR UPDATE` on the same table. The backfill fails. At this point, ~60% of rows have been backfilled; 40% have `NULL` in the new column. 3. **Disagreement surfaces** — Kobe calls for immediate rollback, reasoning that a known-good pre-migration state is safer than a partially-migrated state, especially with an undiagnosed deadlock risk. MJ calls for forward-fix, reasoning that the 60% already migrated are consistent, and that rollback introduces a new, unrehearsed operation (event replay) with its own correctness risk. Both positions are internally coherent. Neither position had been pre-validated by the team — there was no decision framework for this scenario. 4. **Decision: rollback** — The team chooses rollback. The column addition is reverted. The 60% of migrated rows are returned to their pre-migration state. 5. **Event replay design** — Because 4 hours of preference update events accumulated during the migration window and its aftermath, the team must design and execute a replay to bring data current. This replay operation had not been rehearsed. 6. **Replay executes** — 47 minutes to run. During this window, the preference API serves stale (pre-migration) data. Replay completes successfully. No data loss confirmed by Pippen. ## Contributing Factors Each of these factors independently contributed to the outcome. Removing any single one would likely have changed the severity or duration of the incident, but no single factor is "the root cause." **1. Unbatched backfill on a live-traffic table.** The backfill was a single `UPDATE` across 800k rows with no batching. This maximized lock duration and lock scope, creating a large deadlock window with any concurrent transaction. The runbook did not specify a batch size threshold, so Shaq's implementation was compliant with the documented process. **2. Live API traffic during migration window.** The maintenance window did not fully isolate the `user_preferences` table from API traffic. A `SELECT FOR UPDATE` from the live API collided with the backfill. The team's definition of "maintenance window" did not include traffic isolation for the affected table. **3. Semantic ambiguity in the nullable column.** Bird identified that `NULL` in the new column is semantically ambiguous — it could mean "no preference set" or "not yet migrated." This ambiguity was not resolved before migration approval. A partially-backfilled table was therefore not just an operational inconvenience but a domain-correctness problem, which raised the stakes of both the rollback and forward-fix options. **4. No pre-agreed decision framework for partial migration failure.** When the backfill failed, the team had no shared criteria for choosing between rollback and forward-fix. Kobe and MJ both reasoned from first principles under time pressure, arriving at different — and defensible — conclusions. Event replay was unrehearsed.** The rollback path required replaying 4 hours of accumulated events — an operation the team had not practiced.
