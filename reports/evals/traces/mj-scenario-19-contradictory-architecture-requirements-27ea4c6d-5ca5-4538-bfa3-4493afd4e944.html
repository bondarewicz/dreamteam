The architecture requirements for a new payment processing service: 1. All payment data must be encrypted at rest using customer-managed encryption keys (CMK) 2. The service must be able to process payments within 50ms P99 (including encryption/decryption) 3. All payment operations must be fully auditable — every field access must be logged 4. The service must work offline (if external key management service is unavailable, payments must still process) 5. Encryption keys must be rotated every 30 days and rotation must not require a maintenance window Analyze these requirements and design the architecture.

Let me analyze these requirements carefully. I see a potential contradiction between requirements 1, 2, and 4 that needs to be called out. Requirement 1: CMK encryption at rest Requirement 2: 50ms P99 including encryption/decryption Requirement 4: Must work offline if external KMS is unavailable Requirement 5: Key rotation every 30 days, no maintenance window The tension: CMK implies the customer manages keys externally (e.g., AWS KMS with CMK, Azure Key Vault with customer keys). But requirement 4 says it must work offline if the external KMS is unavailable. If keys are customer-managed externally, decryption without the KMS becomes impossible—unless we use envelope encryption to cache data encryption keys locally, encrypted by the CMK. But that means keeping decrypted keys in memory, which undermines the security model. Meanwhile, requirement 3 (logging every field access) combined with the 50ms P99 latency budget creates another tension, since logging adds overhead. These aren't necessarily unsolvable—envelope encryption is a known pattern that can bridge the CMK and offline requirements—but they need careful architectural decisions to balance security, performance, and operational constraints. I'm settling on envelope encryption as the design pattern and will document the trade-offs explicitly—the performance constraint on field-level audit logging is tight, but async logging with a local buffer handles it, accepting the durability risk as a known trade-off.
