Bird — Domain Analysis

Domain analysis for eval runner system
Rule Scores are strictly pass/partial/fail -- no numeric scores, no percentages at scenario level, no half-grades
Rule Every scenario must run in session isolation -- fresh agent instance with no context from prior scenarios
Rule Human scores are authoritative -- system never auto-scores based on text matching or heuristics
Rule Every scored scenario requires a justification -- score without justification must not be persisted
Rule Scenario files are source of truth -- runner must not modify, summarize, or truncate prompts or rubrics
Rule Regression is a score decrease from the immediately prior complete run, not historical best
Rule All 18 scenarios must be scored for a run to be a complete baseline -- partial runs cannot serve as regression baseline
Rule Calibration gap is per-agent: avg stated confidence minus actual pass rate (pass=100%, partial=50%, fail=0%)
Rule Run results are append-only -- once finalized, never retroactively modified
Rule Eval report is standalone HTML -- no external dependencies, renders correctly via file:// protocol
AC Given evals/ directory, when runner initializes, then it discovers all scenarios, validates 4 required fields, reports validation errors
AC Given Bird-01 executed, when Bird-02 begins, then agent has zero context from Bird-01 (session isolation)
AC Given agent output for scenario, when scoring, then runner displays full output + expected_behavior + failure_modes + scoring_rubric and human chooses pass/partial/fail with justification
AC Given user invokes /eval bird, then only Bird's 3 scenarios run, result recorded but not marked as complete baseline
AC Given completed run with 18 scored scenarios, when results saved, then each contains run_id, agent, scenario_id, type, score, justification, confidence_stated, timestamp
AC Given 4 prior complete runs exist, when report generated, then trend table shows date, pass/partial/fail counts, delta from prior run
AC Given current and prior run, when report generated, then any scenario with decreased score is listed as regression
Edge Case Agent produces empty output -- present to human, human decides score (no auto-fail)
Edge Case User aborts mid-run -- partial results saved with incomplete flag, not eligible as regression baseline
Edge Case First-ever run -- no prior baseline, regression section shows 'No prior baseline', no deltas
Edge Case New scenario added between runs -- regression comparison only for scenarios in both runs
Edge Case Two runs same day -- run_id includes HHMM timestamp for uniqueness
MJ — Architecture
88% 16:02 — 16:06
Architecture design for eval runner system
Decision Two components -- commands/eval.md (Claude Code skill for running evals) + scripts/eval-report.sh (bash script for HTML report generation). Separation allows re-generating reports without re-running evals.
Decision JSON-per-run results storage in evals/results/YYYY-MM-DD-HHMM.json -- git-friendly, atomic, no database dependency
Decision Bash script for report gen (not Python) -- matches cast.sh pattern, uses python3 -c for JSON parsing only
Decision Agent spawning via Task tool with subagent_type -- matches existing team.md pattern exactly
Decision Command supports /eval (all), /eval bird (agent), /eval bird scenario-01 (specific), /eval --report (generate dashboard)
Decision AskUserQuestion tool for human scoring interface -- present output + rubric, collect pass/partial/fail + justification
Risk Task tool output capture may be lossy -- large agent outputs could be truncated. Mitigation: test with one scenario first
Risk Scenario file parsing fragility -- YAML-like format with pipe block scalars is not strict YAML. Mitigation: line-by-line parsing keyed on field names
Risk Human fatigue on 18-scenario runs. Mitigation: show progress [3/18], allow mid-run quit with partial save
