# Follow-up: Eval System Improvements (2026-03-24)

Changes to the eval pipeline covering trace capture, trace report generation, grader robustness, and headless eval behavior.

---

## 1. Trace Capture

### What changed

`scripts/eval-run.sh` now invokes the Claude CLI with `--output-format stream-json --verbose` instead of `--output-format json`. This captures the full conversation trace — every intermediate step from prompt to final response.

Each raw output JSON file (`evals/results/raw/<timestamp>/<agent>-<scenario>.json`) now contains a `trace` array alongside the existing fields:

```json
{
  "agent": "bird",
  "scenario_id": "scenario-17-subtle-domain-drift",
  "agent_output": "...",
  "trace": [
    {"type": "system", "model": "claude-opus-4-6", "tools": [...], ...},
    {"type": "assistant", "message": {"content": [{"type": "thinking", ...}]}, ...},
    {"type": "assistant", "message": {"content": [{"type": "text", ...}]}, ...},
    {"type": "result", "duration_ms": 59281, "total_cost_usd": 0.099, ...}
  ],
  "duration_ms": 59281,
  "tokens_used": 2386,
  "cost_usd": 0.099,
  "timestamp": "2026-03-24T11:22:06Z"
}
```

### Trace event types

| Type | What it contains |
|------|-----------------|
| `system` | Init: model, tools, session_id, CLI version, permission mode, MCP servers, context window |
| `assistant` | Message content blocks: `thinking` (chain-of-thought), `text` (response), `tool_use` (tool call with name + input) |
| `user` | Tool results fed back to the model |
| `rate_limit_event` | Rate limit status (skipped in trace pages) |
| `result` | Final summary: stop_reason, turns, duration, tokens, cost, modelUsage, permission_denials |

### NDJSON parsing

The stream-json format emits one JSON object per line (NDJSON). The parser:
- Collects all valid JSON lines into the `trace` array in order
- Skips blank lines and non-JSON lines
- Extracts `agent_output`, `usage`, and `cost` from the last `type=result` event
- Falls back gracefully if no result event is found (crashed/killed agent runs)

### Backward compatibility

Raw output files from before this change (no `trace` key) still work in the report pipeline. No trace link is shown for those scenarios.

## 2. Trace Report Pages

### What changed

`scripts/eval-report.sh` now generates standalone HTML trace pages for every scenario that has trace data.

### File structure

```
reports/evals/
  2026-03-24-eval-report.html          # Main eval report (unchanged format)
  traces/
    bird-scenario-17-subtle-domain-drift-f3fb820b-1e27-4aa7-b7da-9f5c46701077.html
    kobe-scenario-05-sql-injection-a02cea54-5582-4f6a-8178-02dee89197e6.html
    ...
```

### Filenames

Format: `{agent}-{scenario_id}-{uuid4}.html`

UUID4 guarantees no filename collisions across runs, trials, or concurrent generation. For multi-trial runs, trial index is included: `{agent}-{scenario_id}-t{trial}-{uuid4}.html`.

### Trace page content

Each trace page is a standalone HTML file (no external dependencies, viewable offline) showing:

1. **Header** — agent name, scenario ID, score badge, duration, cost
2. **Session info panel** — model, session ID, CLI version, permission mode, tools count, context window, max output, fast mode, MCP servers, permission denials (extracted from system init event)
3. **Step-by-step timeline** — each trace event rendered by type:
   - **User** (blue) — the prompt sent to the agent (injected synthetically since CLI `-p` mode doesn't echo it)
   - **Assistant / Thinking** (purple, italic, left-bordered) — chain-of-thought reasoning
   - **Assistant / Text** (purple) — the agent's response, JSON pretty-printed when applicable
   - **Tool Call** (gold) — tool name + input parameters as key=value pairs
   - **Tool Result** (teal) — tool output with size indicator
   - **Result** (gray) — summary grid with stop_reason, turns, duration, tokens, cache stats, cost
4. **Per-step metadata chips** — input/output tokens, cache hit ratio (green/red), service tier, stop reason
5. **First-failure highlighting** — for failing/partial scenarios, the last assistant step before the result event is highlighted in red to pinpoint where the reasoning went wrong
6. **Show full output toggle** — truncated content (10K cap for text/tool results, 5K for thinking/tool inputs) has a clickable "show full output (N chars)" button that expands to show the complete content
7. **Back-link** — link back to the main eval report

### Linking from scenario cards

Each scenario card in the main eval report has a small "trace" link. Single-trial shows "trace", multi-trial shows "t1 t2 t3" links. No link appears for scenarios without trace data.

### Cleanup

Old trace files are deleted from `reports/evals/traces/` at the start of each report generation. Since traces are regenerated from raw data, old files are always stale.

## 3. Grader Fix: Markdown Fence Handling

### The bug

Agents (especially Shaq) sometimes wrap their JSON output in markdown code fences:

```
Here is the output:

```json
{"implementation_summary": {...}, "acceptance_criteria_coverage": [...]}
```​
```

The grader's JSON extraction logic (`_find_json_in_output` and the `json_field` grader) used a brace-depth scanner that failed on fenced output — it would find a nested object instead of the root object, causing field-presence checks to fail incorrectly.

### The fix

Introduced a shared `_extract_json(output)` helper in `scripts/eval-graders.sh` that tries three strategies in order:

1. **Direct parse** — `json.loads(output.strip())`
2. **Fence strip** — extract content from `` ```json ... ``` `` or `` ``` ... ``` `` fences via regex, then parse
3. **Brace scanner** — fallback depth-based scanner for JSON embedded in prose

Both `json_valid` and `json_field` graders now use this shared helper, eliminating duplicated logic.

### Impact

Re-scoring the same agent outputs with the fixed grader improved pass rate from 68.8% to 71.2% (+3 scenarios correctly graded).

## 4. Headless Eval: Plan Mode Prevention

### The problem

Shaq's agent definition includes plan-mode instructions ("MUST use plan mode before writing any code") and uses the `opusplan` model variant. In headless eval runs (`claude -p`), the agent would present a plan, ask "Ready to implement — approve to proceed?", and stop with `stop_reason: end_turn`. No approval comes, so the agent never produces structured output.

### The fix

Added `--append-system-prompt` to the Claude CLI invocation in `eval-run.sh`:

```
EVAL MODE: You are running in a headless evaluation. Do NOT enter plan mode.
Do NOT call EnterPlanMode. Do NOT wait for approval. Execute the task directly
and produce your complete final output immediately.
```

This override:
- Applies only during eval runs (not interactive sessions)
- Appends after the agent's system prompt, giving it recency advantage
- Is harmless for agents that don't use plan mode (no-op instruction)
- Does not modify any agent definition files

### Why not other approaches

| Option | Why not |
|--------|---------|
| `--permission-mode dontAsk` | Controls tool permissions, not agent-initiated plan mode |
| `--model opus` override | Changes what you're evaluating — undermines eval validity |
| Conditional logic in agent .md | Pollutes agent definitions with eval concerns |
